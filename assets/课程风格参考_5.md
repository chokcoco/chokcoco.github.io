# 第一節 什麼是 Embedding

[![English](https://img.shields.io/badge/Language-English-blue)](./01_what_is_embedding_en.md)

在前面的章節中，我們已經完成了資料載入與文字分塊。接下來，RAG 系統要做的事情是：把每一個 chunk 轉成可以被電腦快速比較的格式。

這個格式就是 **Embedding 向量**。

Embedding 可以理解成一種「語意座標」。它會把文字轉成一串數字，例如：

```text
"什麼是 RAG？"
-> [0.021, -0.183, 0.447, ...]
```

這串數字不是隨機產生的，而是由 embedding model 根據文字語意計算出來的。語意相近的文字，轉成向量後會在向量空間中比較接近；語意差異大的文字，向量距離通常會比較遠。

在 RAG 裡，Embedding 是 retrieval 的基礎。沒有 embedding，系統只能用關鍵字比對；有了 embedding，系統就可以做語意搜尋。

## 一、為什麼 RAG 需要 Embedding

假設資料庫裡有一段文字：

```text
RAG 會先從外部知識庫找出相關內容，再交給 LLM 生成答案。
```

使用者問：

```text
大型語言模型要怎麼引用自己的資料？
```

這兩句話沒有完全相同的關鍵字，但語意上其實很接近。如果只用傳統關鍵字搜尋，可能找不到這段內容；但如果使用 embedding，模型可以判斷它們都和「用外部資料輔助 LLM 回答」有關。

這就是 embedding 在 RAG 中最重要的作用：**把字面不同、但語意相近的內容找出來**。

RAG 的 retrieval 流程可以簡化成：

```text
文件 chunks -> embedding model -> chunk vectors -> vector store

使用者問題 -> embedding model -> query vector -> 相似度搜尋 -> 找回相關 chunks
```

重點是：文件與使用者問題必須使用同一個 embedding model 轉成向量，這樣兩邊的向量才會落在同一個語意空間中，才能互相比較。

## 二、Embedding 的基本概念

### 2.1 向量是什麼

向量可以想成是一組數字，用來表示某段文字的語意特徵。

例如一個 embedding model 可能會把一段文字轉成 1024 維向量：

```text
[0.13, -0.02, 0.88, ..., -0.41]
```

這裡的每一個數字單獨看通常沒有明確的人類語意，但整個向量組合起來，就能表示這段文字在模型眼中的語意位置。

在 RAG 中，我們通常不會對整份文件產生一個向量，而是對每個 chunk 產生一個向量。原因是整份文件通常太長、主題太多，壓成單一向量後語意會變得模糊。

### 2.2 向量空間

Embedding model 會把文字放進一個高維向量空間中。

可以用下圖來想像：

![Embedding 高維向量空間示意圖](./images/high-dimensional%20embedding%20space.png)

上圖只是概念示意。實際的 embedding 維度通常是幾百到幾千維，不是人類能直接看見的二維或三維平面。

語意相近的文字會靠近，例如「RAG 是什麼」和「檢索增強生成」；語意無關的文字會比較遠，例如「今天晚餐吃什麼」。

## 三、如何比較兩個向量是否相似

Embedding 產生向量後，RAG 系統需要判斷 query vector 和 chunk vector 哪些最相似。常見的相似度計算方式有三種。

### 3.1 Cosine Similarity

Cosine similarity 會比較兩個向量的方向是否接近。

值越接近 `1`，代表兩個向量方向越相似，語意通常也越接近。

在語意檢索中，cosine similarity 很常見，因為它重視方向，而不是向量長度。

### 3.2 Dot Product

Dot product 會計算兩個向量的內積。

如果向量已經正規化，dot product 的效果會接近 cosine similarity。因此很多 RAG 範例會搭配：

```python
encode_kwargs={"normalize_embeddings": True}
```

這代表產生 embedding 時，會把向量長度正規化，讓後續相似度比較更穩定。

### 3.3 Euclidean Distance

Euclidean distance 會計算兩個向量之間的直線距離。

距離越小，代表越相似。不過在文字語意檢索中，cosine similarity 和 dot product 更常被使用。

## 四、Embedding 在 RAG Pipeline 的位置

一個基本 RAG pipeline 通常會包含：

```text
Data Loading
-> Text Chunking
-> Embedding
-> Vector Store
-> Retrieval
-> LLM Generation
```

Embedding 位於 chunking 和 vector store 中間。

### 4.1 建立索引階段

建立知識庫時，流程通常是：

1. 載入文件。
2. 把文件切成 chunks。
3. 使用 embedding model 把每個 chunk 轉成向量。
4. 把 chunk 原文、metadata、embedding vector 存進 vector store。

這一步通常是離線處理，只有當文件新增、刪除或更新時才需要重新執行。

### 4.2 查詢階段

使用者提問時，流程通常是：

1. 使用同一個 embedding model 把問題轉成 query vector。
2. 到 vector store 中搜尋最相似的 chunk vectors。
3. 取回 Top-K chunks。
4. 把 chunks 放進 prompt，交給 LLM 生成回答。

所以 embedding 的品質會直接影響 RAG 找回來的內容。如果 retrieval 找錯資料，後面的 LLM 就算很強，也可能根據錯誤上下文回答。

## 五、Embedding Model 和 Context Window

Embedding model 也有輸入長度限制，通常稱為 **max sequence length** 或 **context window**。

以本課程使用的 `BAAI/bge-m3` 為例，它可以處理最長 **8192 tokens** 的輸入，並且支援多語言、多粒度與多種檢索能力。

但這不代表 chunk 就應該切到 8192 tokens。

原因是 embedding 本質上是一種語意壓縮。當一個 chunk 太長，裡面混合太多主題時，整段文字最後仍然會被壓成一個向量。這個向量會同時代表許多概念，導致檢索時不夠精準。

例如一個 chunk 同時包含：

```text
資料載入、OCR、文字分塊、embedding、vector database、prompt engineering
```

當使用者只問「embedding 是什麼？」時，這個 chunk 可能包含答案，但它的向量也混入了其他主題，語意焦點會被稀釋。

因此 chunk 大小需要在兩件事之間取得平衡：

| 考量 | 說明 |
| --- | --- |
| 太小 | 容易失去上下文，單一 chunk 資訊不足 |
| 太大 | 主題混雜，embedding 向量變得模糊 |
| 適中 | 保留足夠上下文，同時讓語意主題集中 |

## 六、Dense、Sparse 與 Hybrid Retrieval

在 RAG 檢索中，常見方法不只一種。

### 6.1 Dense Retrieval

Dense retrieval 是一般最常見的 embedding retrieval。

它會把文字轉成密集向量，也就是大多數維度都有數值：

```text
[0.13, -0.02, 0.88, 0.41, ...]
```

優點是能理解語意相似，即使文字表面不一樣，也可能找得到相關內容。

限制是如果資料裡有很精確的關鍵字、型號、錯誤碼、法條編號，dense retrieval 有時不如關鍵字搜尋穩定。

### 6.2 Sparse Retrieval

Sparse retrieval 比較接近傳統關鍵字搜尋，例如 BM25。

它重視詞是否出現、出現頻率、詞的重要性。對於精確詞彙很有用，例如：

```text
API_KEY
PDFInfoNotInstalledError
chunk_overlap
BAAI/bge-m3
```

但 sparse retrieval 對語意改寫比較不敏感。如果使用者換一種說法，它可能找不到。

### 6.3 Hybrid Retrieval

Hybrid retrieval 會結合 dense retrieval 和 sparse retrieval。

簡單來說：

| 方法 | 擅長 |
| --- | --- |
| Dense retrieval | 語意相似、同義改寫、概念查詢 |
| Sparse retrieval | 精確關鍵字、專有名詞、錯誤碼 |
| Hybrid retrieval | 同時兼顧語意與關鍵字 |

`BAAI/bge-m3` 的特色之一是支援 dense retrieval、sparse retrieval 和 multi-vector retrieval，因此很適合拿來理解現代 RAG 檢索方法。

## 七、在 LangChain 中使用 Embedding

本課程前面的 LangChain 範例使用：

```python
from langchain_huggingface import HuggingFaceEmbeddings

embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-m3",
    model_kwargs={"device": "cpu"},
    encode_kwargs={"normalize_embeddings": True},
)
```

這段程式會建立一個 Hugging Face embedding model，後續可以用它把文件與問題轉成向量。

### 7.1 重要參數

| 參數 | 範例 | 說明 |
| --- | --- | --- |
| `model_name` | `"BAAI/bge-m3"` | 指定要使用的 embedding model |
| `model_kwargs` | `{"device": "cpu"}` | 傳給模型初始化階段的設定 |
| `device` | `"cpu"` | 指定模型在哪個裝置上執行，也可以是 `"cuda"` |
| `encode_kwargs` | `{"normalize_embeddings": True}` | 傳給文字編碼階段的設定 |
| `normalize_embeddings` | `True` | 是否把輸出的向量正規化 |

`model_kwargs` 是模型載入時的設定，例如使用 CPU 或 GPU。

`encode_kwargs` 是實際把文字轉成向量時的設定，例如是否正規化、batch size、precision 等。

### 7.2 `embed_query`

`embed_query` 用來把單一使用者問題轉成向量：

```python
query_vector = embeddings.embed_query("什麼是 RAG？")
```

它的輸入是一個字串，輸出是一個向量。

通常 retrieval 階段會使用這個方法，因為使用者一次通常只問一個問題。

### 7.3 `embed_documents`

`embed_documents` 用來把多段文件文字轉成向量：

```python
doc_vectors = embeddings.embed_documents([
    "RAG 是檢索增強生成。",
    "Embedding 會把文字轉成向量。",
])
```

它的輸入是一個字串 list，輸出是一組向量 list。

建立 vector store 時，通常會對 chunks 使用這類方法。

## 八、如何選擇 Embedding Model

選 embedding model 時，不應該只看排行榜分數。RAG 專案更需要考慮自己的資料型態與部署條件。

### 8.1 Hugging Face MTEB Leaderboard

如果想比較不同 embedding model 的表現，可以參考 Hugging Face 上的 [MTEB Leaderboard](https://huggingface.co/spaces/mteb/leaderboard)。

![Hugging Face MTEB Leaderboard](./images/leaderboard.png)

MTEB 是 Massive Text Embedding Benchmark 的縮寫，它會用多種任務評估 embedding model，例如 retrieval、classification、clustering、reranking、semantic textual similarity 等。這比只看單一測試集更接近真實應用，因為 embedding 不只會被用在 RAG，也常被用在分類、聚類與相似度比對。

看 leaderboard 時，可以特別注意幾個欄位：

| 欄位 | 說明 |
| --- | --- |
| Model | 模型名稱，可以點進去看模型卡、授權與使用方式 |
| Mean (Task) | 多個任務類型的平均分數，可以快速比較整體表現 |
| Retrieval | 和 RAG 最相關，因為 RAG 核心任務是找回相關文件 |
| Active Parameters | 實際參與推論的參數量，會影響記憶體需求、部署成本與推論速度 |
| Embedding Dimensions | 向量包含的數值數量。維度越高，通常能表示更細緻的語意，但也會增加 vector store 的儲存空間與搜尋成本 |
| Max Tokens | 模型可接受的最大輸入長度，會影響 chunk 設計 |

不過，leaderboard 只能當作選型參考，不應該直接等於最佳選擇。

實務上建議先用 leaderboard 篩出候選模型，再拿自己的資料做小型測試。尤其是中文 RAG、特定領域文件、錯誤碼查詢、法規條文、產品手冊這類資料，排行榜分數高不一定代表在你的資料上效果最好。

### 8.2 評估面向

選 embedding model 時，可以從以下面向評估：

| 評估面向 | 說明 |
| --- | --- |
| 語言支援 | 中文、英文或多語言資料要選對應模型 |
| Max sequence length | 是否能處理你的 chunk 長度 |
| 向量維度 | 維度越高通常資訊越多，但儲存與搜尋成本也更高 |
| 檢索任務表現 | 要看 retrieval 任務，而不是只看分類或摘要表現 |
| 推論速度 | 本地 CPU、GPU 或 API 都會影響速度 |
| 部署成本 | 模型越大，硬體或 API 成本可能越高 |
| 領域適配 | 法律、醫療、金融、程式碼等專業資料可能需要特化模型 |

本課程使用 `BAAI/bge-m3` 的原因是它支援多語言、長文本輸入，也適合中英文混合的 RAG 教學情境。

## 九、常見問題

### 9.1 Chunk 越長越好嗎

不是。

Chunk 太長會讓一個向量包含太多主題，retrieval 時可能變得不精準。即使 embedding model 支援長輸入，也應該讓每個 chunk 盡量聚焦在單一主題。

### 9.2 可以用不同 embedding model 建索引和查詢嗎

不建議。

文件 chunks 和使用者 query 應該使用同一個 embedding model。不同模型產生的向量空間不同，直接比較通常沒有意義。

### 9.3 更新文件後需要重新 embedding 嗎

需要。

只要文件內容改變，原本的 embedding vector 就不再代表新的內容。更新資料庫時，通常要重新切 chunk、重新 embedding，並更新 vector store。

### 9.4 Embedding 可以取代 LLM 嗎

不行。

Embedding 負責把文字轉成向量，主要用途是檢索、分類、聚類、相似度比較。LLM 則負責理解上下文並生成回答。兩者在 RAG 裡扮演不同角色。

## 十、本節重點整理

1. Embedding 是把文字轉成向量的技術。
2. RAG 使用 embedding 來做語意檢索。
3. 文件 chunks 和使用者問題要用同一個 embedding model。
4. 相似度常用 cosine similarity、dot product 或 Euclidean distance。
5. Chunk 太大會讓向量語意變模糊，chunk 太小則可能缺少上下文。
6. Dense retrieval 擅長語意搜尋，sparse retrieval 擅長精確關鍵字，hybrid retrieval 可以結合兩者。
7. 選擇 embedding model 時，要考慮語言、長度限制、維度、速度、成本與領域需求。

## 十一、練習

可以嘗試用本課程的 `BAAI/bge-m3` 模型觀察不同句子的向量相似度：

```python
from langchain_huggingface import HuggingFaceEmbeddings

embeddings = HuggingFaceEmbeddings(
    model_name="BAAI/bge-m3",
    model_kwargs={"device": "cpu"},
    encode_kwargs={"normalize_embeddings": True},
)

sentences = [
    "RAG 會從外部知識庫找資料。",
    "檢索增強生成可以讓 LLM 使用外部文件回答問題。",
    "今天晚餐想吃義大利麵。",
]

vectors = embeddings.embed_documents(sentences)

print(len(vectors))
print(len(vectors[0]))
```

可以思考：

1. 前兩句的語意是否比較接近？
2. 第三句和前兩句是否比較無關？
3. 如果把非常多主題塞進同一個 chunk，產生的向量會代表什麼？
