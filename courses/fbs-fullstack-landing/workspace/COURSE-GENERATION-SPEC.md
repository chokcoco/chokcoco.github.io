## 状态

- 阶段：课前准备完成 / 标杆章已通过质检
- 日期：2026-07-16
- 质检决定：`ready_for_batch_generation`

## 课程结构

课程负责人已确认采用以下 6 个主题模块：

1. 前端语言及标准库基础
2. Web 前端开发
3. Web 前端进阶
4. 后端语言及标准库基础
5. Web 后端开发
6. Web 后端进阶

FBS 业务不单列百科模块，只作为工程章节中的真实语境和最终纵向实践。

## 章节数量与体量

- 每模块 4-8 章。
- 当前蓝图共 40 章：6 + 7 + 6 + 7 + 7 + 7。
- 所有正式知识章节的有效字符最低为 9000 Unicode 字符，不设最高字符限制。
- 9000 是内容完整度的最低门槛，不是写作目标；达到 9000 不代表通过，也不得因接近某个数值而压缩必要解释。
- 章节长度以“跨端学习者能够读懂真实代码、完成受控修改并解释验证证据”为止。主题存在多个相互独立的学习成果时应拆章，而不是把无关内容堆进一章。
- 有效字符统计排除代码围栏标记、HTML 标签、纯导航、参考链接 URL 和重复模板文本；具体统计脚本在正文阶段加入。
- 重复解释、长代码粘贴、表格复述和无学习价值的背景铺陈不计作有效增量，也不得用于凑足最低字符数。
- 字符不足时必须回到本章学习路径，补齐缺失的概念、机制、边界、对比、案例或练习，并把内容放在它所依赖的正文位置。独立练习、章末自检或小结之后不得继续追加新的主体知识。
- 面向另一端零基础学习者时，章节标题或主线首次出现的技术名词要先解释含义、在架构中解决的问题，再说明常见同类方案及当前仓库方案的主要取舍；对比用于建立选择边界，不扩展成仓库未采用技术的操作教程。

## 交付格式

沿用 `ai-course-studio` 的静态课程结构：

- Markdown 为正文事实源。
- HTML 为学生阅读产物。
- `course.json` 管理课程与章节元数据。
- 配套速查、练习和项目资产放在 `reference/` 与 `labs/` 或仓库既有等价目录。
- 交付核心为 Markdown 事实源和 HTML 学生阅读产物，不生成讲师 PPT。
- 交付目标为公司内部可访问的静态课程站点；不要求公开版本或额外脱敏副本。

## 教学路径与风格

- 基础讲师风格：耐心教练型。
- 工程操作章：耐心教练型 + 实践带练型。
- 架构、并发、事务、缓存、消息与安全章：耐心教练型 + 严谨技术型。
- 故障章：耐心教练型 + 案例分析型。
- 不叠加三个以上风格标签。

语言章主要使用“语言基础与标准库”类型；工程章主要使用“框架/工具”类型；架构与诊断章主要使用“原理机制/故障诊断”类型；模块收束和最终项目使用“综合实践”类型。环境安装内容作为工程章中的辅助模块，不单独扩成通用工具课。每章的主模板路径、辅助模块、插入位置、服务目标和证据以 `CHAPTER-TEMPLATE-MAP.md` 为强制约束。

## 代码与业务证据

- 每个非语言基础章节必须引用至少 2 个仓库证据，其中至少 1 个为代码、配置或 manifest。
- 代码片段尽量来自指定仓库；为教学缩减时明确标为“缩减示例”，不得声称可直接替换生产代码。
- 声称可运行的命令、示例与测试必须实际执行，或在质量报告中标为未验证。
- FBS 业务断言优先使用 `fbs-kb` 中 truth status 较高且有 code evidence 的主题；`unknown` 与 `partially_unresolved` 不写成确定事实。
- `fbs-kb` 仅作为作者侧业务来源。学生正文不介绍该仓库的目录、索引、生成流程、证据状态机制或知识库工程实践。
- 最终实践优先选择同时具有前端、后端、知识库和流程材料证据的主题。

## 练习与验收

每章选择一个主教学任务：概念讲解、操作引导、观察实验、案例分析、故障诊断、渐进练习、项目交付或阶段复习。练习必须直接验证学习成果。

推荐证据组合：

- 语言章：预测输出 → 修改短代码 → 阅读仓库片段。
- 前端工程章：定位文件 → 修改现有页面/路由/API → lint/type-check/build 或页面行为。
- 后端工程章：追踪接口 → 修改 DTO/handler/service/repository → 单测或接口响应。
- 进阶章：日志/指标/配置/消息证据 → 建立假设 → 最小验证 → 回归。
- 最终实践：需求拆解、跨仓影响清单、实现、测试、联调、构建与开发交接。发版只用一句话提示遵循团队通用流程，不设置发版操作或回退材料验收。

## 来源与保密

- 所有仓库、PPT、PDF、截图、Confluence 和内部平台链接均视为公司内部资料。
- 与操作或理解直接相关的内部内容 URL 可以出现在学生页面。内部材料的创作路由、作者 workspace、本机路径和生成过程不进入学生正文。
- 课程不制作公开脱敏副本；测试账号、token、密钥和真实 PII 仍不得固化在 Markdown、HTML、配置或练习资产中。
- 内部来源只在 workspace 和质量报告中保存断言映射。
- 如课程最终仅限受控内网，仍需对账号、密钥、token、PII 和个人路径做披露扫描。

## 质量闸门

正文阶段按以下顺序执行：

1. 生成 Markdown 与 HTML。
2. 对学生可见中文执行 `humanizer-zh`，目标不低于 45/50。
3. 运行 `node .agents/skills/teach-generator/scripts/validate-course.mjs courses/fbs-fullstack-landing`。
4. 按 `references/course-rubric.md` 生成第二阶段报告。
5. 浏览器检查导航、响应式、代码块、Mermaid 和交互；运行课程中声称可运行的代码与命令。

长课程按连续 3-4 章一批完成闭环。任何内容修改后，旧质检报告失效。

## 已确认基线

- 学习者具备 Git、命令行和至少一种编程语言经验。
- 只支持 macOS + 公司开发环境，不提供 Windows 分支。
- 以研发自学为主，不设置单节时长上限；每章至少 9000 有效字符且无上限，篇幅由讲透学习成果和完成证据决定。
- 六个仓库（3 前端 + 3 后端）的 release 分支（2026-07-20）均为批准事实基线。
- 主案例为入库请求/ASN 小型纵向变更。
- 课程不会要求学习者在 live 环境执行写操作。
