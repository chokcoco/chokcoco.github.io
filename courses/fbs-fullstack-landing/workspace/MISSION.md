# FBS 前后端转全栈 Landing 课程任务定义

## 课程为什么存在

这门课服务于 FBS 团队内部的前后端转全栈过程。它不以系统讲完前端或后端知识为目标，而是帮助已有一端研发经验的同学尽快进入另一端的真实工程：能启动仓库、读懂主要代码路径，完成一个纵向需求的分析、前后端实现、联调、测试、构建与开发交接。

## 主要受众

- 前端研发：已有 JavaScript/TypeScript、浏览器和前端工程经验，但 Go、服务端框架、数据库、中间件经验较弱。
- 后端研发：已有 Go/服务端开发经验，但 JavaScript/TypeScript、React、Vue、浏览器和前端工程经验较弱。
- 次级受众：测试、技术负责人和新加入 FBS 的全栈研发，可按需走仓库导览与端到端实践路线。

课程不假设“会一端就自然理解另一端”。首次出现的跨端术语、工具、代码结构和操作步骤都要补齐最小前置。

## 完成课程后的可观察成果

学习者应能：

1. 说明 3 个前端仓库与 3 个后端仓库的职责、运行方式、技术栈差异和主要边界。
2. 在 React 16 Portal、Vue 3 MMF、React 18 MMF/远端组件中定位页面、路由、状态、API、权限、i18n 与构建入口。
3. 在 Go 服务中定位入口、Chassis 路由、中间件、Wire 注入、应用/领域/仓储层、配置、任务和测试。
4. 沿一个真实 FBS 业务流程追踪浏览器、前端请求、后端接口、数据持久化、异步任务和可观测信号。
5. 在现有仓库模式下完成一个小型纵向需求，并提供 lint/type-check/test/build、接口响应、页面行为和开发交接证据。
6. 遇到常见启动、联调、权限、配置、缓存、消息或任务问题时，能按证据缩小范围，而不是跨层猜测。

## 能力分级

| 等级 | 新获得的能力 | 常见卡点 | 进入下一等级的标准 |
| --- | --- | --- | --- |
| L1 读懂 | 能识别另一端语言、工具和目录中的核心对象 | 把框架语法当作语言语法；只记文件名，不理解执行顺序 | 能解释一段最小代码的输入、状态与输出 |
| L2 跟改 | 能按仓库现有模式修改单层功能 | 不知道应复用哪个封装；遗漏权限、i18n、错误码或 Wire | 能完成受控修改并通过最小验证 |
| L3 串联 | 能追踪一次跨层请求和数据变化 | 只看正常路径；忽略宿主、配置、事务、异步边界 | 能画出真实链路并用日志、响应或测试验证 |
| L4 独立交付 | 能完成小型纵向需求的分析、实现、联调与验证 | 跨仓影响评估不足；验证停留在本地成功 | 能交付实现、测试、回归、构建与 reviewer 可复核的开发交接证据 |

## 核心路线

课程采用双入口、同一终点：

- 后端转前端：前端语言与 Web 标准 → 三类前端工程 → 微前端与交付 → Go 快速阅读旁路 → 后端链路导览 → 纵向交付。
- 前端转后端：Go 语言与标准库 → 三类后端工程 → 数据、任务与基础设施 → 前端快速阅读旁路 → 联调与页面验证 → 纵向交付。

所有学习者最终都完成同一条端到端实践。课程负责人已确认以“入库请求/ASN 的小型变更”为主案例，因为它在前端、后端、业务参考和培训材料中都有较完整证据。

## 内容边界

课程只讲以下范围：

- 8 个指定仓库：课程工程、3 个前端仓库、3 个后端仓库和 FBS 知识库。
- `assets/fbs/` 下的 4 份补充材料、用户给出的基础设施截图，以及这些来源明确链接的 Confluence 页面。
- 代码仓库当前出现且服务于 Landing 的语言、框架、工具、中间件、数据库、基础设施、测试和交付流程。

课程明确不讲：

- 与指定仓库无关的通用前后端知识大全。
- 仓库未使用的前端框架、后端框架、数据库或云原生方案。
- 3PL、Data-fix 等补充材料已标为“不重要”且不在指定仓库范围的系统。
- 深入讲解 ISC、WMS、SLS、SPX、Finance 等外部域的内部实现；只保留 FBS 调用边界和联调所需信息。
- `fbs-kb` 的仓库结构、生成流程、知识组织方式和维护机制。该仓库只提供业务参考材料，不是学员需要 Landing 的开发仓库。
- 把内部账号、密钥、域名、个人路径、未公开链接或 PII 写入学生可见正文。

## 教学与验收原则

- 语言基础以“能读懂并做小改动”为标准，不从零穷举语法。
- 其余模块必须绑定实际仓库文件、真实运行入口或真实业务链路。
- 每章至少有一种完成证据：解释、预测、代码修改、运行结果、测试、日志诊断、影响分析或交付物。
- 综合实践按纵向切片推进，每个里程碑保持可运行、可验证，并能说明兼容边界。
- 前后端发版属于团队通用研发流程，不纳入本课程授课和实操验收；相关章节最多提示“完成开发交接后遵循团队通用发版流程”。
- 内部材料用于作者取证。与操作、规范或业务理解直接相关的内部内容 URL 可以出现在学生页面；课程生成过程、作者 workspace、本机路径、账号、token、密钥和真实 PII 不进入正文。

## 已确认的学习与交付方式

- 课程固定为 6 个模块，FBS 业务语境穿插在工程章节和最终项目中，不单独设置业务知识模块。
- 主要学习方式为研发自学；不设置单节时长或字符上限。所有正式知识章节至少 9000 Unicode 有效字符，并以讲透学习成果、能够完成真实仓库任务为准。
- 运行环境只支持 macOS 与公司开发环境。
- 六个仓库的 release 分支（2026-07-20）是本轮课程的批准事实基线。
- 最终核心产物为 Markdown 与 HTML，不生成讲师 PPT。
- 课程只在公司内部流通，不要求额外制作公开脱敏版本；安全凭据和真实 PII 仍不写入课程资产。

## 当前停止条件

课前准备完成后先生成一前一后两份标杆章。标杆章通过 Markdown、HTML、Humanizer 和三阶段质检后，再按连续 3-4 章一个批次生成；任何批次未通过不得标记为完成。
