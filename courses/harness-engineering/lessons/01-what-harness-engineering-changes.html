Harness Engineering 到底改变了什么？

Harness Engineering 把工程关注点从“让模型生成一次结果”扩展到“让智能体在一套可检查、可恢复的工作系统中持续完成任务”。

先看一个合成案例

下面的情节由几类常见失败拼成，不对应某家真实企业。

一个用了多年的前端仓库有五个应用、几个共享组件包，还有两套路由和权限写法。产品经理提出一个小需求：

在排班记录页面增加“导出原因”字段，并确保没有导出权限的人看不到它。

工程师把需求交给编码智能体。十分钟后，智能体给出了一份看上去很完整的修改：页面上多了一列，接口类型也更新了，代码还能通过格式检查。

问题出在更深处。

  • 它改的是旧版页面，新版页面由另一个路由入口加载。
  • 它直接判断用户角色，没有复用仓库现有的权限服务。
  • 它只跑了当前文件的检查，没有执行应用测试。
  • 它看到命令退出码为 0，便宣布任务完成。

继续修改提示词或许能救回这一次任务。下次换一个页面，问题还会回来。仓库没有告诉智能体该从哪里进入、哪些事实可信、哪些目录不能碰，也没有用验收测试定义“真的完成”。

这就是本章要处理的问题：智能体失败时，我们究竟该修提示词、换模型，还是修它工作的那套系统？

Harness Engineering 在这里指什么

这里把 Harness Engineering（驭缰工程）定义为一类工程工作：围绕智能体设计清楚的意图、可读的环境、合适的工具、行为边界、持久状态和反馈回路，使任务能够执行、检查、恢复，并在失败后得到改进。

不同团队对 harness 的边界划分并不完全一致，这个定义主要用于分析工程系统，不是一项强制标准。OpenAI 在 2026 年 2 月公开了一项内部实验：一个团队让 Codex 生成产品代码、测试、CI、文档和内部工具，人类把精力转到设计环境、明确意图和构建反馈回路。端到端自治依赖该仓库的具体结构与工具，不能在缺少同等投入时直接外推。

“工程师不再写代码”是那项实验采用的约束，不是这门课对所有团队的要求。现实里，工程师仍然会写代码、查问题、做设计和审核。变化发生在主工作对象上：过去我们主要盯着某一次实现，现在还要维护一套让后续任务更容易做对的工作系统。

可以把这套系统拆成六个观察面。它们会在后续章节逐一展开。

观察面它要回答的问题在开头案例中的缺口
意图目标、范围和完成标准是什么？“增加字段”没有写清新版页面、权限行为和验收场景
环境与知识智能体应该读什么，哪些事实可信？新旧路由并存，却没有入口地图和事实来源
工具它能通过什么接口读取、修改和验证？只会改文件，没有可靠的页面定位与权限查询入口
边界哪些行为允许，哪些需要审批或禁止？可以绕过权限封装，也能修改错误模块
状态进度、决策和证据保存在哪里？退出码被当成唯一完成记录
反馈谁检查结果，失败怎样送回执行过程？没有相关测试，也没有按验收条件复查页面

Harness 的价值不在于把这六栏都写满。真正的判断标准是：它们能否让智能体在当前任务上少猜一点，在出错时更早暴露，并给出下一步可用的信号。

这和“写一段好提示词”差在哪儿

提示词很重要。任务目标含糊时，再完整的工程环境也不知道该往哪里走。问题是，提示词只能承载部分信息，而且常常只对当前会话有效。

做法常见产物适合解决容易留下的问题
提示词改进一段更清楚的任务说明消除当次任务的歧义规则难复用，执行结果未必可检查
工作流设计固定步骤、角色或审批节点让重复任务按顺序流转遇到新情况时可能缺少判断和工具能力
Harness Engineering任务契约、仓库知识、工具、权限、状态、测试与反馈让智能体在真实环境里持续完成并纠错建设和维护有成本，需要围绕高价值失败逐步投入

三者可以一起使用。好的 harness 会把提示词放在合适的位置，也可能包含固定工作流。区别在于，团队开始对“智能体怎样工作、怎样知道自己错了”承担工程责任。

一个简单例子：

  • 提示词写着“不要修改共享包”。这是指导。
  • 权限策略让智能体没有共享包的写权限。这是边界。
  • 检查脚本发现 diff 触碰共享包就失败。这是反馈。
  • 失败信息同时列出受影响消费者和建议处理方式，智能体可以据此修正。这才形成可用的闭环。

指导仍然有用，但能机械检查的规则通常更稳定。OpenAI 的案例也经历了类似变化：团队先维护仓库文档，随后把重要架构约束做成自定义 lint 和结构测试，并让错误信息包含修复线索。

工程师的工作发生了哪些移动

传统研发里，工程师已经在做需求澄清、架构、测试和运维。Harness Engineering 没有凭空发明这些工作。它改变的是使用对象和反馈速度：原本写给人看的约定，要让智能体能发现；原本靠评审经验拦住的问题，要尽量变成可执行检查；原本散落在聊天和脑子里的判断，要进入可维护的事实源。

从“告诉它怎么写”转到“说明怎样算完成”

“请写一个高质量的导出功能”无法检验。下面这组条件更有用：

  • 只修改新版排班页面及其局部 API 类型。
  • 无权限用户看不到入口，也不能通过直接请求绕过限制。
  • 导出文件包含新增字段，空值显示为空字符串。
  • 相关单元测试、权限测试和一次页面冒烟检查通过。
  • 如果必须修改共享包，停止执行并说明影响范围。

这组条件仍然需要后续细化，但已经可以转成测试、权限检查和人工验收表。第 3 章会把这种表达整理成任务契约。

从“把所有资料塞进去”转到“让它知道去哪里找”

长文档会占用上下文，也会变旧。OpenAI 的案例曾尝试用一个很大的 AGENTS.md 承载所有指导，后来改成短入口加结构化文档目录。短入口负责导航，详细事实放在可以分层读取和检查的位置。

对已有仓库，这意味着先建立地图：应用、路由、模块、共享依赖、测试入口和风险边界。对新仓库，则可以从第一天就设计稳定目录、架构边界和事实源。两种起点不一样，目的相同：减少无依据的探索。

从“人肉发现问题”转到“让问题尽早回到循环”

如果每次都等到人工评审才发现智能体选错模块，团队只是把编码时间换成了审查时间。更好的处理方式，是在任务早期检查路由映射，在修改前确认文件边界，在提交前运行相关测试。

人仍然负责判断。只是那些重复、明确、可以计算的检查不必每次靠人提醒。人的注意力留给需求取舍、风险接受和例外决策。

两种仓库起点，建设方法并不相同

接下来会沿两条案例线观察同一套原则。

第一步：改造已有企业仓库

已有仓库的麻烦通常来自事实分散、历史模式冲突和共享依赖难追踪，补一个 AGENTS.md 解决不了这些问题。此时不能直接套一份漂亮模板。先做定位与收敛：

  1. 找出真实入口和当前有效实现。
  2. 标出允许修改、需要告知和禁止修改的范围。
  3. 为当前任务裁剪最小上下文，而不是总结整个仓库。
  4. 把现有测试、命令和人工验收接进反馈过程。
  5. 当证据不足或影响面过大时可靠停止。

存量仓库的定位结论需要可追溯：事实来自哪个文件、是否仍覆盖当前版本、当前任务究竟需要哪些上下文。当证据不足或影响范围失控时，系统应当停止并说明原因，而不是继续猜。

第二步：从零建设新仓库

新仓库没有历史包袱，但也没有可复用经验。团队需要尽早做出几项选择：目录和模块怎样分层，依赖方向如何检查，任务和架构决策放在哪里，智能体使用哪些命令，完成一次修改要留下哪些证据。

这里的风险是过度设计。团队可能在第一行业务代码出现前，就建立几十份规则和一套没人维护的平台。更稳妥的方式是先给当前任务准备最小可用 harness，等真实失败出现后再增加规则。

后面的实践会让两条案例线使用同一项业务需求。我们会比较：哪些组件可以共用，哪些必须尊重仓库历史。

四个角色会怎样工作

Harness Engineering 不只属于“会写智能体框架的人”。一套能工作的 harness 需要不同角色把隐性判断写清楚。

角色过去容易停留在在 harness 中增加的工作
产品与业务需求描述和验收会议给出可检查的行为、边界、优先级和例外
研发与架构实现、代码评审、口头约定把架构边界、命令和局部规则做成智能体能发现的入口与检查
测试与质量功能完成后的验证提前设计失败样本、验收信号和回归范围
平台、安全与管理工具采购、权限配置和过程治理管理沙箱、审批、审计、成本和自治等级

零基础学习者不需要马上实现这些机制。先学会提问就够了：任务完成的证据在哪里？智能体改错地方时，系统能否发现？如果连续工作两小时，它怎样记住已经做过什么？

常见误解

“模型越强，就越不需要 harness”

模型变强会扩大可完成任务的范围，也会扩大它能够采取的行动。真实工程仍有仓库规则、权限、状态和验收条件。更强的模型可能减少一些指导，却不会自动知道团队没写出来的业务边界。

“把规则都写进系统提示词就够了”

提示词里的规则可能被忽略、互相冲突或随上下文增长而失去作用。路径权限、类型检查、测试和审批能提供更明确的信号。规则放在哪里，取决于它是否需要解释、执行或审计。

“Harness 的目标是完全去掉人”

高风险变更、需求冲突和跨团队边界通常需要人判断。可靠停止本身就是能力：系统知道什么时候缺少证据，并把问题交给合适的人。

“一次把平台建全，后面自然会顺”

Harness 会随模型、工具和仓库一起变化。没有真实任务和失败样本，团队很难知道哪些规则值得维护。先处理高频返工，再逐步扩大覆盖，通常更省力。

本章练习：诊断失败发生在哪一层

下面是一段简化运行记录。先不要设计解决方案，只标出缺口属于意图、环境与知识、工具、边界、状态还是反馈。

请回答：

  1. 这次任务缺少哪些可验证的完成条件？
  2. 智能体为什么容易选错文件？
  3. 哪一道边界检查可以在修改前阻止它直接改共享函数？
  4. 哪些反馈能在 09:11 之前暴露问题？
  5. 如果任务中断，下一次运行至少要保留什么状态？

参考答案与判分依据

答案不要求使用完全相同的术语。能够指出证据和因果关系即可。

问题合格答案应包含
1明确目标页面、金额单位或格式、移动端与后台范围、不得破坏退款页，以及相应验收场景
2搜索结果没有业务入口、模块所有权、路由或调用关系，智能体只能按词面猜测
3共享代码修改前的 notice/approval gate，或把共享目录移出默认写权限；最好同时列出消费者
4目标页面测试、退款页回归测试、调用点分析、移动端冒烟检查；只回答“多跑测试”不算完整
5已确认的目标模块、修改文件、做过的判断、测试结果、未解决风险和下一步

五项中答对四项，并能说明至少一项如何进入执行循环，可以认为掌握本章目标。只重复“提示词不够好”还不够，因为它没有定位可复用的系统改进。

本章小结

Harness Engineering 把关注点从一次回答扩展到智能体工作的整个系统。模型仍然重要，提示词也重要，但真实任务还需要可发现的事实、可执行的工具、明确边界、持久状态和能回到执行过程的反馈。

已有仓库先处理历史与边界，新仓库可以从第一天安排结构和验证。两种路径都不追求规则越多越好。下一章会继续往里拆：模型负责什么，agent loop 怎样运行，完整 harness 又比一个循环多了什么。
