模型、agent loop 与 harness 是什么关系？

模型负责当前判断，agent loop 连接判断与行动，harness 则管理循环所需的上下文、工具、权限、状态和完成条件。

同一个模型，为什么表现像两个不同的同事

把同一项任务交给两套系统：

找到登录失败的原因，修复它，运行相关测试，并报告仍未覆盖的风险。

系统甲只能和模型对话。模型根据你贴出的报错推测原因，给出几段代码建议。它无法确认仓库里是否真的有那些文件，也不知道测试命令是什么。

系统乙会读取仓库，搜索调用链，运行测试，查看失败输出，修改文件后再次验证。遇到需要访问生产凭证的步骤时，它暂停并请求批准。会话中断后，它还能从保存的进度继续。

两套系统可以使用同一个模型。差别来自模型外面的运行机制。

这一章会把几个经常混在一起的词拆开：model（模型）、agent（智能体）、agent loop（智能体循环）和 agent harness（智能体运行支架）。这些词在不同产品和文章里边界略有差异，我们采用一套方便分析工程系统的工作定义。

四个词，四种职责

概念这里采用的定义它负责什么它通常不独立负责什么
model接收当前输入并生成输出的推理组件理解文本、比较方案、提出下一步动作或给出答案真正执行命令、长期保存项目状态、落实权限策略
agent围绕目标运行，能观察环境并借助工具采取行动的实体决定下一步、使用工具、根据结果继续自动保证工具安全、事实新鲜或验收完整
agent loop在模型判断和工具执行之间反复运行的控制循环组装输入、发起推理、执行动作、收集观察结果、决定继续或结束覆盖整个产品的认证、持久化、审计和客户端协议
agent harness把模型、循环、工具、上下文、状态、权限、验证和生命周期组织起来的执行系统为智能体提供可控的工作条件和反馈代替业务目标、组织判断和所有人工审核

可以先记一个关系：模型在循环里做判断，agent 通过循环行动，harness 拥有并约束这条循环。

这句话是便于学习的压缩表达，不是行业标准定义。真正分析产品时，要继续问：谁组装上下文？谁执行工具？谁保存状态？谁能停止任务？这些职责可能分布在不同进程和服务中。

一次 agent loop 怎样运行

以 Codex 为例，系统把用户输入加入提示并调用模型；模型可以给出最终回答，也可以请求工具调用。工具结果会加入后续输入，模型再次判断。循环持续到模型不再请求工具，转而给用户输出消息。

工程系统还会在工具调用前后加入权限、状态和验证。下面这张图采用课程的通用视角，不等同于某个产品的内部实现。

flowchart TD
  A["任务与当前状态"] --> B["Harness 组装上下文"]
  B --> C["模型判断下一步"]
  C --> D{"需要采取行动吗？"}
  D -- "否" --> E["检查终止条件"]
  E -- "满足" --> F["返回结果与证据"]
  E -- "不满足" --> B
  D -- "是" --> G["权限与策略检查"]
  G -- "拒绝或需人工判断" --> H["停止或请求批准"]
  G -- "允许" --> I["执行工具"]
  I --> J["产生观察结果"]
  J --> K["更新上下文与持久状态"]
  K --> B

按顺序看一遍：

  1. Harness 取出任务说明、仓库规则、已有进度和可用工具，组成当前输入。
  2. 模型判断下一步。它可能请求读取文件、运行测试，也可能认为已经可以回答。
  3. 工具调用先经过权限或策略检查。高风险动作可能需要人批准。
  4. 工具执行后产生观察结果，例如文件内容、测试失败或页面截图。
  5. Harness 把必要结果送回上下文，并将需要跨步骤保留的信息写入状态。
  6. 模型根据新证据继续。系统检查完成、失败和停止条件，直到本轮结束。

这里有一个容易忽略的点：工具输出不等于事实已经被正确使用。测试日志太长可能被截断，错误信息可能只写“失败”，搜索结果也可能混入旧文件。Harness 需要控制观察结果的格式、大小和可信来源。

模型做判断，harness 管条件

把职责分开，有助于定位问题。

更接近模型的问题

  • 已经提供了正确文件和错误日志，模型仍把调用关系理解反了。
  • 工具说明很清楚，模型连续选择不适合当前目标的工具。
  • 约束没有冲突，模型却生成了语义明显错误的实现。

这类问题可能通过更合适的模型、推理设置、示例或任务分解得到改善。

更接近 harness 的问题

  • 正确的架构文档存在，但系统没有把入口或查找方式提供给 agent。
  • 模型请求运行测试，工具却只能返回“命令已完成”，没有退出码和失败详情。
  • 系统允许 agent 读取生产密钥，且没有审批或审计记录。
  • 会话中断后，进度只留在已经丢失的聊天上下文里。
  • agent 说“完成了”，系统没有核对验收条件。

这时单纯换模型可能暂时绕开症状，系统缺口仍然存在。

现实里两类原因会叠在一起。例如模型先选错工具，harness 又没有参数校验；模型忘记一个条件，系统也没有验收测试。诊断时不要急着找唯一责任方，先确定哪个环节本可以更早暴露错误。

上下文为什么会成为工程问题

每次模型判断都只能看到当前输入。对话历史、工具调用和工具结果不断加入后，输入会增长；模型又有上下文窗口上限。Codex 会在达到阈值后压缩对话，为后续判断腾出空间。

压缩能腾出空间，但不能替团队决定什么必须永久保存。下面两类信息应该分开：

  • 当前推理上下文：为了下一步判断临时装入的任务片段、文件、日志和历史。
  • 持久项目状态：已确认的需求、决定、进度、产物、测试证据、未解决风险和交接信息。

如果一项架构决定只存在于聊天第 120 轮，它可能在压缩后失真，也难以被下一个会话复核。把稳定事实写回仓库，把临时信息按需装入上下文，通常比保存一条无限增长的聊天记录可靠。

第 10 章会专门处理长时任务。本章只需要记住：上下文是一次判断能看到什么，状态是系统需要保留什么。两者有重叠，生命周期不同。

完整 harness 为什么比 loop 大

Agent loop 解释了“模型怎样通过工具反复行动”。一个可用产品还要处理会话、权限和运行环境。

Codex 的完整体验除了核心循环，还处理 thread 的创建、恢复、分叉、归档和历史持久化，加载配置与认证，在沙箱中执行文件或 shell 工具，并接入 MCP server 与 skills。App Server 再用双向协议把这些能力提供给不同客户端。

这些是 Codex 的产品设计，不是通用 harness 必须照抄的组件和协议。我们从中提取的是职责问题：

职责一个完整系统要作出的决定
生命周期任务、turn、会话何时创建、暂停、恢复、归档
工具执行命令在哪里运行，能读写哪些资源，输出怎样返回
权限与批准哪些动作自动允许，哪些动作需要人确认
状态与持久化哪些历史必须保留，客户端断线后怎样继续
扩展新工具、skills 或业务接口如何注册和约束
观察与审计如何查看进度、diff、失败、成本和最终证据

一个简单的本地脚本也可以有 harness。它不需要 App Server，只要明确管理当前任务所需的循环、工具、状态和验证。反过来，采用了复杂 agent 框架也不代表 harness 已经可靠；如果完成条件和失败反馈仍然含糊，框架只会更快地执行含糊任务。

把开头的登录任务放进循环

先看已有企业仓库。

任务要求修复登录失败。Harness 在第一次模型判断前提供：当前生效的登录模块、认证服务入口、允许修改路径、相关测试命令、生产凭证禁用规则。模型读取错误日志，调用搜索工具定位到 token 解析逻辑，修改前通过边界检查。测试失败后，完整断言和堆栈被送回循环。模型修复实现并再次运行测试。最后，系统根据验收表检查“错误可复现、修复后通过、未改认证协议、风险已报告”，然后允许结束。

换成新仓库，任务不变，准备工作不同。仓库还没有登录模块、测试命令和权限封装。Harness 需要先完成初始化：确定目录和依赖边界，创建认证接口契约，建立密钥处理规则，准备最小测试入口。模型随后才进入功能实现循环。

两条路径都使用 agent loop。已有仓库的难点是找到真实事实并尊重历史；新仓库的难点是尽早建立可以延续的结构。

通过症状定位 harness 缺口

运行症状优先检查原因
agent 总在错误目录修改上下文入口、仓库地图、目标定位模型可能没有得到当前有效实现的位置
agent 会读文件，却不运行测试工具描述、任务契约、终止条件系统可能把“生成修改”误当成“完成任务”
测试失败后重复同一种修改观察结果质量、状态、失败分类错误信号没有提供可行动的新证据
每次新会话都从头探索持久状态、事实源、交接协议进度没有离开临时上下文
小任务经常触碰共享代码修改边界、审批、影响分析默认权限过宽，缺少风险门禁
agent 很早宣布完成验收检查、终止条件、独立审校最终消息被误当成完成证据

这张表不是自动诊断器。它的作用是把“agent 不靠谱”拆成可以检查的部件。

常见误区

把 agent 当成模型的另一个名字

模型生成输出，agent 还要在环境中行动。同一个模型放进不同工具、权限和反馈条件里，会表现出不同的工程能力。比较系统时，应同时记录模型与 harness 配置。

认为工具越多，agent 越强

工具会扩大行动空间，也会增加选择错误、权限越界和上下文噪声。工具接口需要明确用途、参数、失败方式和返回证据。当前任务用不到的工具可以不暴露。

把“回复完成”当成终止条件

模型输出最终消息，只能说明这一轮停止请求工具。工程任务是否完成，要看验收标准、测试和风险说明。Harness 可以在模型准备结束时再运行完成检查，未通过就把结果送回循环。

把聊天历史当成可靠项目状态

聊天记录适合保留对话轨迹，不适合单独承担长期事实源。可复核的决定、进度和证据应进入稳定文件或状态存储，并标注来源和新鲜度。

本章练习一：给循环补上缺失环节

阅读下面的轨迹：

请在不换模型的前提下，提出四项 harness 改进。每项必须写明：放在哪个环节、返回什么信号、怎样改变下一步。

参考答案

以下是一个合格方案，不是唯一答案。

  1. 在组装上下文时加入订单模块入口、仓库筛选的现有相似实现和允许修改目录，减少 126 个搜索结果带来的猜测。
  2. 在工具调用前加入共享包修改门禁。若请求修改 shared/query-builder.js，先列出消费者并要求批准；未批准则返回局部替代方案提示。
  3. 让测试工具返回命令、退出码、失败测试名和关键日志。只有退出码为 0 且命中相关测试，才能形成“测试通过”的证据。
  4. 在结束前检查验收条件：筛选参数、无筛选兼容、权限或数据范围、相关回归测试。缺少任一证据就把缺口送回循环。

四项中至少覆盖上下文、边界、观察结果和终止条件，并说明反馈如何回到下一次判断，即可通过。

本章练习二：画出你自己的 agent loop

选择一项熟悉的工作。它可以是修改代码、生成测试用例、整理数据，也可以是制作周报。画出一条最小循环，至少标出：

  • 任务输入和完成条件
  • 模型每轮能看到的上下文
  • 可调用工具及权限检查
  • 工具返回的观察结果
  • 需要持久保存的状态
  • 成功、失败和转人工条件

自检清单

完成后逐项检查：

  • 图中是否存在从工具结果回到下一次判断的箭头？
  • “测试通过”“报告正确”之类的结论是否有可观察证据？
  • 是否区分了当前上下文和跨会话状态？
  • 高风险工具是否有拒绝、审批或沙箱？
  • 即使模型想结束，系统是否还会核对完成条件？

五项全部可以在图中找到，说明这条循环已经具备最小分析价值。后续章节会继续给它添加任务契约、仓库知识和质量反馈。

选择团队正在使用的编码智能体，不需要阅读源码。根据实际界面、日志和文档填写下表，不确定的地方写“未知”。

问题观察到的实现证据风险或待确认项
谁组装模型输入？
工具有哪些，如何注册？
shell 或文件操作在哪里执行？
哪些动作需要批准？
会话与任务状态保存在哪里？
怎样处理上下文增长？

这张表的价值在“未知”栏。团队无法回答的职责，往往也是系统出现异常后最难排查的地方。

本章小结

模型负责根据当前输入做判断。Agent 通过工具在环境中行动，agent loop 把判断、行动和观察连接起来。Harness 则管理这条循环所依赖的上下文、工具、权限、状态、验证和生命周期。

遇到失败时，先沿循环寻找信号在哪里丢失：模型是否看到了正确事实，工具是否返回可用证据，风险动作是否受控，状态能否跨会话保留，终止是否经过验收。这样做比笼统地说“模型不够聪明”更容易得到下一项工程改进。
