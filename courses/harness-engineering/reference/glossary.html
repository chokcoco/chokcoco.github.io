Harness Engineering 术语表

快速查阅课程中的核心术语与使用边界。

本页收录全课程会反复使用的术语。定义服务于本课程，涉及具体产品时会另行注明。

术语课程中的含义容易混淆的地方
Harness Engineering（驭缰工程）围绕智能体设计意图、环境、工具、边界、状态和反馈，让工作可以执行、检查、恢复与改进不是某个厂商的产品名，也不等于完全无人开发
model（模型）接收输入并生成输出的推理组件模型本身通常不负责真正执行命令、保存工程状态或判断权限
agent（智能体）在目标驱动下，能借助工具观察环境并采取行动的运行实体“会对话的模型”未必已经具备可靠行动能力
agent loop（智能体循环）模型判断、请求工具、接收观察结果并继续或结束的一轮轮循环一次用户请求可能包含多次模型调用和工具调用
agent harness（智能体运行支架）组织模型、循环、工具、权限、上下文、状态、验证和终止的执行系统agent loop 是核心机制之一，但完整 harness 通常更大
context（上下文）当前一次模型判断可以看到的任务、历史、文档、代码和工具结果仓库里存在的信息不一定已经进入当前上下文
state（状态）需要跨步骤或跨会话保留的进度、决策、产物和环境信息聊天记录只是状态的一部分，不能代替项目事实源
tool（工具）智能体可调用的文件、终端、浏览器、数据库或业务系统接口工具可用不代表工具被正确授权或输出可验证
observation（观察结果）工具执行后返回给智能体的事实，例如测试失败、文件内容或页面状态观察结果需要足够具体，智能体才能据此修正
feedback loop（反馈回路）产出、检测、反馈、修正和回归验证组成的循环只有检测、没有把结果送回执行过程，不算闭环
guardrail（护栏）限制行为范围或要求确认的规则、权限、沙箱与审批护栏负责控制风险，不能代替正确的任务定义
termination condition（终止条件）规定任务何时成功结束、何时失败停止、何时交给人的条件“智能体说完成了”不是充分的完成条件
task contract（任务契约）把目标、范围、不变量、验收、失败处理和停止条件写成可版本化的工作说明不是要求智能体照抄实现步骤的长提示词
invariant（不变量）实现可以变化，但不能破坏的业务或架构规则“高质量”不是不变量；“金额使用整数分”可以检查
reliable stop（可靠停止）证据不足、来源冲突或风险越界时，保存证据并停止猜测停止需要说明阻塞点和下一安全动作
progressive disclosure（渐进式披露）先给短入口，再按任务需要读取局部事实和源码不是一次把所有资料塞入上下文
Context Pack（上下文包）为一次任务裁剪的最小事实、入口、验证和边界集合它随任务变化，不是永久仓库总结
freshness（新鲜度）一条事实是否仍覆盖当前版本和任务历史文档存在不代表当前仍有效
idempotency key（幂等键）标识一次业务意图，使相同请求可以安全重放同一键代表不同意图时应报冲突
eval（评测）用任务、环境、样本和证据衡量模型与 harness 的共同表现不等同于单元测试，也不能只看最终答案
rubric（评价量规）用维度、等级和可观察证据评价难以精确断言的产物不是一套跨行业统一的自动检测标准
maker-checker生成者产出，独立检查者依据契约和证据审校的分工checker 可以是脚本、模型或人工组合
autonomy level（自治等级）规定智能体可独立执行到什么范围、哪里需要人工确认不是模型能力排行榜，需结合风险和恢复能力
golden principles（黄金原则）团队长期维护的少量、可检查工程约束规则应有所有者和证据，不宜无限增加
garbage collection（持续清理）定期删除过期指令、坏模式、永久例外和无效工具不是一次性大重构

术语使用约定

  • 本课程用“驭缰工程”指 Harness Engineering，用“智能体运行支架”解释 agent harness。
  • 初次学习时可以保留英文。企业落地时应先统一团队术语，避免同一个“harness”同时指运行时、测试框架和 Harness.io 产品。
  • 文中的 Codex、Claude Code 等名称只表示案例所处的产品环境。通用原则不依赖某个模型或厂商。
