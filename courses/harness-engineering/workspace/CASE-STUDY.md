# 课程案例设计：星桥商城

本文件用于作者侧保持跨章一致，不进入学生页面导航。

## 定位

- 虚构品牌：星桥商城（Starbridge Market）
- 仓库类型：包含两个 React 应用、一个共享促销包和一个 Spring Boot 服务的单仓库
- 学习用途：同时展示存量仓库改造和新仓库建设
- 数据政策：商品、人员、订单、运行记录和故障均为虚构；只借鉴公开工程资料中的问题类型

## 功能一：优惠叠加与价格解释

顾客购买两件单价 399 元的商品，可能同时命中会员折扣、满减券和免运费活动。旧仓库有三套不一致的规则：前台本地计算、共享包旧实现、后端结算实现。任务要求统一组合规则，并返回逐项价格明细。

验收重点：

- 相同输入在前端预览和后端结算得到相同结果
- 会员折扣与订单券是否组合由显式策略决定
- 已参加特价的商品不重复应用商品级折扣
- 价格使用整数分，禁止浮点累计
- 输出原价、每项优惠、运费和应付金额
- 共享包修改前必须列出消费者并触发 notice

## 功能二：库存预占与超时释放

顾客提交订单时先预占库存，支付成功后正式占用；支付取消或 15 分钟超时后释放。客户端在网络超时后可能重试，因此预占接口必须使用幂等键。多仓场景优先从能完整履约的仓库预占。

验收重点：

- 同一幂等键重复请求不会重复扣减可售库存
- 同一幂等键配不同请求体时拒绝执行
- 支付成功后的 claim 只能发生一次
- 取消和超时释放不会把库存加回两次
- 库存不足时不产生部分预占残留
- 关键状态转换、失败原因和人工处理入口可观察

## 公开资料与改写边界

- Shopify 公开资料用于确认折扣组合存在产品级、订单级和运费级等不同类别，组合需要显式规则
- Shopify 2026 年库存预占文章用于确认 reserve、claim、多仓、并发和超卖/少卖是现实工程问题
- Stripe 与 Amazon 公开资料用于确认网络失败下的幂等键、重试和语义冲突处理
- 课程不复刻上述公司的源码、数据库方案、吞吐数字或组织流程

## 技术栈

- React 19.2 + Vite 8.1，两个轻量单页应用
- Node.js 22，纯 JavaScript 促销引擎和无依赖验证脚本
- Java 21 + Spring Boot 4.1 + Maven，单体模块化服务和内存型教学存储
- 默认不要求 Docker、Redis、MySQL 或云账号，避免环境准备盖过课程目标
