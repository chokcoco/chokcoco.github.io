## 逐维证据

### 学习目标与覆盖

通过证据：

- `lessons/01-why-ai-code-review.md` 的“Code Review 的六个阶段”“四类审查者怎样分工”“AI CR 的职责边界”覆盖发展阶段、Shift-Left、机器与人工边界。
- `lessons/02-prompt-to-harness.md` 的“一次请求怎样穿过系统”“五个核心数据对象”“allowed_next_step”覆盖 Session/Batch/File/Issue、MCP 两工具和 `0→1→2→2.x→3`。
- `lessons/03-metrics-adoption-recall-f1.md` 的“当前采纳率的完整过滤链”“当前召回率是怎样构造的”逐项解释 score、状态、有效性、Bug 与角色过滤。

问题证据：第 2 章只预告 PendingTask 与 MemoryRule，没有在本批展开远端任务和记忆生命周期。这与蓝图一致，分别留给第 8 章及第 5/7 章，不构成遗漏。

### 事实与证据

通过证据：

- 第 1 章把 Bacchelli/Bird 的现代 Code Review 结果、Google 工程准则与当前 AI CR 产品能力分开陈述，没有用工程准则冒充效果实验。
- 第 2 章明确约 800–1200 是当前案例的主要理想范围，并指出旧材料 800–1500 的版本差异；Verifier 默认值与热配置的事实边界也被保留。
- 第 3 章准确写出正式分母 `[1,4,0,5]`、自动状态 2/3 与部分状态 60–69 的排除，并指出查询纳入 `is_valid=1 or null` 而非笼统“只统计已验证有效”。

问题证据：GitHub/GitLab 产品能力会随版本变化。正文已使用官方文档且不写具体价格和承诺；`RESOURCES.md` 保存访问日期 2026-07-15。后续整课发布前应复查产品章节。

### 受众适配与脚手架

通过证据：

- 第 1 章在首次使用时解释 CR、MR、Shift-Left 和静态分析，并以十个脱敏问题场景建立共同经验。
- 第 2 章先解释 Prompt 的职责过载，再引入 Harness；MCP 先讲 Host–Client–Server，再映射案例工具。
- 第 3 章从混淆矩阵进入公式，再进入真实状态与 SQL 管道，没有要求学习者预先掌握统计学。

问题证据：第 2、3 章均超过 11,000 有效字符，术语和状态密度高。HTML 已提供粘性目录、图、表和分节；浏览器阶段仍需验证移动端目录和宽表滚动体验。

### 解释深度与连贯性

通过证据：

- 第 2 章用“模型自我报告不是完成证据”贯穿 Setup、Planner、MCP、Agent、Verifier 和状态持久化，正常路径与六个失败点互相验证。
- 第 3 章不止给公式，还用 12 条混合状态样本说明过滤顺序，用两种 Recall 口径计算两个 F1，并用 34+44 定向样本反例解释选择偏差。
- 跨章衔接明确：第 1 章提出噪音/遗漏，第 2 章解决可靠执行，第 3 章建立后续优化的判定语言。

问题证据：第 2 章“状态预测练习”标题下面还包含分批、schema 和并发保护等机制模块，标题范围略窄。内容顺序仍然连贯，判为 minor，可在整课编辑时改名为“完成证明与状态预测”。

### 练习与反馈

通过证据：

- 第 1 章“路线检查”要求按证据把问题交给 lint、静态分析、AI 或人工。
- 第 2 章给出四个状态情形，学习者需判断谁能推进状态以及缺失结果如何恢复。
- 第 3 章“复算任务”要求重算 `is_valid=null`、Bug 去重和定向样本偏差，能直接暴露是否理解分母。

问题证据：当前没有可下载 CSV 或可运行计算器。课程规格明确实践以思维启发为主，本批次的表格可手工复算，故不构成 major；后续可在第 7 章简化 Harness 支架中加入同一指标函数。

### 语言与教师感

通过证据：

- 章首只保留预计学习时间和一句话总结。
- 句子围绕具体代码、状态、数字和失败条件，不使用模糊的“行业专家认为”。
- 没有装饰性粗体、表情符号、通用积极结论或创作过程信息。

问题证据：第 1 章有少量“不只是/不是……而是……”用于真实概念边界；第 2、3 章也有少量否定式区分。未形成连续排比，保留比机械替换更自然。
