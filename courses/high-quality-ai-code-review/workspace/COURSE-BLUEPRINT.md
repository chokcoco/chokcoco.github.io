# 课程蓝图：如何实现高质量的 AI Code Review

状态：课程蓝图已确认 v0.2；第 1–3 章进入生成。

## 课程主线

课程围绕一个工程问题推进：怎样让 AI Code Review 不只是“偶尔给出一条好建议”，而是在真实规模的仓库中持续给出值得采纳、尽量少漏报、可以被量化和复盘的结果？

主线采用以下因果链：

`人工 CR 的覆盖与成本问题 → 简单 AI CR 的上下文与长任务失效 → Harness/状态机 → 指标口径 → 采纳率改造 → 召回率改造 → 数据闭环 → 远端服务化`

## 能力等级与进入条件

### L1：看懂问题与边界（第 1 章）

- 新能力：区分人工 CR、静态规则和 AI CR 各自擅长的任务。
- 常见卡点：把 AI CR 说成“替代所有人工判断”，或把静态检查与语义审查混为一谈。
- 进入条件：能根据问题类型选择适合的审查手段。

### L2：建立系统与度量模型（第 2–3 章）

- 新能力：解释一次审查请求的完整链路，并能复算关键指标。
- 常见卡点：把架构图当成实现证据；忽略状态分母和不可判定样本。
- 进入条件：能说清 Session/Batch/Issue 的关系，以及采纳率、召回率的当前统计口径。

### L3：诊断并提升本地审查质量（第 4–7 章）

- 新能力：针对低采纳和低召回设计不同的实验、控制点和数据回流。
- 常见卡点：只加 Prompt；用问题数量替代质量；把方案阈值当成普适标准。
- 进入条件：能把一个失败现象映射到规则、上下文、工作流、验证或数据层。

### L4：服务化与迁移（第 8–9 章）

- 新能力：设计异步远端 AICR，处理 Git、并发、幂等、回调和可观测性，并形成落地路线。
- 常见卡点：认为“把本地命令放到服务器”就完成了服务化。
- 完成标准：提交一份有边界、有指标、有失败路径和验收证据的 AICR 方案。

## 核心 20%–30%

全课最高权重放在四个对象上：

- 外部 Harness 与 `allowed_next_step` 状态机。
- 采纳率/召回率的实现口径和数据偏差。
- 批次分配 + 验证 + 扩展检查构成的召回率工程。
- 远端任务的异步执行、Git 工作区复用和回调可观测性。

这些内容会在架构图、代码观察、诊断练习和结课实践中重复迁移，但不重复解释同一段定义。

## 章节计划

### 第 1 章：为什么今天需要 AI Code Review

- 核心问题：AI CR 要替代人工 CR 的哪一部分，又必须保留哪些人工判断？
- 学习路径：从低频、高成本、重复性人工评论出发，对比静态规则、远端 Diff 审查、本地上下文审查和工程化 AICR，最后划定适用边界。
- 主正文类型：课程开篇。
- 主教学任务：案例分析；辅助任务为概念分类。
- 风格：严谨技术型 + 简洁专业型。
- 必须覆盖：Code Review 发展阶段、Shift-Left、低级/重复问题、AI 生码质量门、人工与 AI 的职责边界。
- 完成证据：给 8–10 个问题场景分类，说明应交给 lint、AI、人工或组合处理。
- 关键图：Code Review 演进时间线；问题类型 × 审查方式矩阵；课程路线图。
- 主要来源：FE/ALL PPT、公开 Code Review 一手资料（待补）。
- 最低体量：7,000 有效字符。

### 第 2 章：从 Prompt 到 Harness：整体架构与核心思路

- 核心问题：为什么更长的 Prompt 不能独立保证一条复杂审查流程可靠执行？
- 学习路径：沿一次真实请求穿过 CLI/入口、Setup、Server、MCP、Agent、结果提交和汇总；再用职责边界解释 Planner、TL、Bridge 和 Executor。
- 主正文类型：架构链路。
- 主教学任务：架构讲解；辅助任务为源码追踪。
- 风格：严谨技术型 + 案例分析型。
- 必须覆盖：Prompt-only 与 Harness、Session/Batch/File/Issue、`allowed_next_step`、`0→1→2→2.x→3`、MCP 两个核心工具、链式提示词和失败点。
- 借用模块：机制类“真实请求/数据流”；综合实践类“源码观察点”。
- 完成证据：学习者根据给定状态预测下一步，并指出哪个组件有权推进状态。
- 关键图：四层职责图；一次审查时序图；Session 状态机；核心数据对象关系图。
- 主要来源：核心方案 PDF、FE PPT 14–15 页、`setup.ts`、`processor.ts`、MCP 工具和数据模型。
- 最低体量：11,000 有效字符。

### 第 3 章：先建立度量：采纳率、召回率与 F1-Score

- 核心问题：什么数据才足以证明 AICR 变好了？
- 学习路径：先定义 TP/FP/FN 与人工反馈，再拆当前代码中的采纳状态、有效问题过滤、召回分母和 Bug 归属，最后用样本复算并暴露偏差。
- 主正文类型：机制解释与实验验证。
- 主教学任务：观察实验；辅助任务为数据计算。
- 风格：严谨技术型。
- 必须覆盖：Precision/采纳率、Recall/召回率、F1；人工/Feature Flow/自动检测/部分采纳/无法判断；`score >= 4`、有效问题；含 Bug 与不含 Bug 的召回；FE/BE Bug 归属边界。
- 完成证据：复算一个混合状态样本，并解释为何自动 2/3、部分采纳 60–69 或 PM/unknown Bug 未进入当前正式口径。
- 关键图：指标混淆矩阵；采纳状态流；召回分母树；统计偏差边界图。
- 主要来源：评估 PDF、`crShiftLeftIssue.ts`、`adoption-status.ts`、`crAnalytics.ts`、`bugSync.ts`。
- 最低体量：11,000 有效字符。

### 第 4 章：采纳率低：问题为什么“看起来对，却没人采用”

- 核心问题：低采纳率是模型不够强，还是审查系统给错了上下文和标准？
- 学习路径：从拒绝问题样本建立假设，分别检查团队规则缺失、上下文不足、建议不可执行、Diff 定位错误、重复建议和建议价值不足。
- 主正文类型：故障诊断。
- 主教学任务：案例分析。
- 风格：严谨技术型 + 案例分析型。
- 必须覆盖：通用规则与项目规则、分类/评分/intent、代码上下文、recheck、重复过滤、拒绝类型、采纳检测局限。
- 完成证据：对一组低采纳案例进行归因，并为每个假设指定可观察证据。
- 关键图：采纳率根因树；一次误报的证据链；规则与上下文注入图。
- 主要来源：ALL PPT 4 页、FE PPT 16–17 页、`promptBuilder.ts`、`filterChecker.ts`、`adoptionChecker.ts`。
- 最低体量：11,000 有效字符。

### 第 5 章：提升采纳率：规则、上下文、复核与负向记忆

- 核心问题：怎样建立一个能持续减少噪音、又不把有效问题一并过滤的闭环？
- 学习路径：先建立规则层次和上下文包，再加入 recheck、历史问题去重和负向记忆；最后用自动采纳检测与人工反馈验证改变。
- 主正文类型：综合应用实践。
- 主教学任务：问题驱动改造。
- 风格：严谨技术型 + 实践带练型。
- 必须覆盖：通用/项目规则、语义分组、关联上下文、recheck、negative memory、过滤索引、LCS 自动采纳检测、短代码冲撞和部分采纳状态。
- 借用模块：机制类“LCS 与短代码边界”；综合实践类“改造前后指标”。
- 完成证据：设计一条负向反馈到下次审查的链路，说明如何避免过拟合个人偏好。
- 关键图：采纳率改造分层图；负向记忆生命周期；LCS 判断流程；改造实验表。
- 主要来源：`memoryDistiller.ts`、`memoryChecker.ts`、`filterChecker.ts`、`adoptionChecker.ts`、ALL PPT 4/6 页。
- 最低体量：11,000 有效字符。

### 第 6 章：召回率低：大型任务中 AI 为什么会“偷懒”

- 核心问题：为什么模型能审好前几批文件，却在长任务后段出现漏检和概括性输出？
- 学习路径：从约 200 文件、约 1.6 万行的观察出发，依次验证“只加强 Prompt”“让 AI 自己管理五步流程”“直接拆成多次人工对话”三种方案的边界，再得到外部控制结论。
- 主正文类型：机制解释与故障诊断。
- 主教学任务：观察实验。
- 风格：严谨技术型 + 案例分析型。
- 必须覆盖：长任务衰减的可观察表现、上下文与指令遵循限制、三次失败尝试、AI 既当运动员又当裁判的问题、事实与推断边界。
- 完成证据：根据一段审查轨迹识别“战略性缩短/合并处理/遗漏步骤”，并设计能区分模型能力与流程问题的实验。
- 关键图：任务进程 × 输出质量曲线；三次失败实验；Prompt-only 与外部 Harness 对比。
- 主要来源：核心方案 PDF、FE PPT 18–23 页；公开长上下文研究（待补）。
- 最低体量：11,000 有效字符。

### 第 7 章：提升召回率：分批、状态机、质检与数据回流

- 核心问题：怎样让系统在不相信 AI 自我报告的前提下，推动它完成每一批审查？
- 学习路径：把大任务分批，后端只下发当前任务；提交结果后依次执行工作量验证和扩展检查；失败重审或熔断；最后把漏报沉淀为 positive/missed memory 和测试集。
- 主正文类型：综合应用实践。
- 主教学任务：项目交付；辅助任务为故障诊断。
- 风格：严谨技术型 + 实践带练型。
- 必须覆盖：三阶段批次算法、800–1200 当前主要分批范围、期望时间/问题密度/字段质量、当前默认配置可能关闭质量门、扩展检查顺序及条件、重试与熔断、positive/missed memory、Benchmark/动态测试集。
- 完成证据：给一个大 MR 生成批次计划和验收状态机，并解释阈值如何通过仓库基线校准。
- 关键图：批次算法；验证决策树；扩展检查状态机；Good/Bad/Missed Case 数据闭环。
- 主要来源：`batchAllocator.ts`、`verifier.ts`、`utils/config.ts`、`extensionChecker.ts`、`memoryDistiller.ts`、核心方案 PDF、FE PPT 31 页。
- 最低体量：11,000 有效字符。

### 第 8 章：远端 AI Code Review 云服务

- 核心问题：把本地 AICR 变成可供 CI、Web 平台和生码 Agent 调用的服务，需要补齐哪些工程能力？
- 学习路径：先对比本地与远端的触发/环境/生命周期，再沿“接入—入队—准备—审查—回调”追踪任务；深入 Mirror + Worktree、并发、幂等复用、恢复、监控和失败上报。
- 主正文类型：架构链路。
- 主教学任务：案例分析；辅助任务为已有系统改造。
- 风格：严谨技术型 + 案例分析型。
- 必须覆盖：同步接入/异步执行/环境解耦、PendingTask、content_hash、CAS 软认领、同 repo+branch 串行、p-limit、Session 复用、Mirror + Worktree、外部回调、监控与恢复、服务化解决的原本限制。
- 完成证据：设计一个外部生码平台接入协议，标出幂等键、终态、回调、超时、重试和观测指标。
- 关键图：本地/远端对比；五阶段远端流水线；Git 工作区拓扑；任务状态与恢复；监控四象限。
- 主要来源：远端服务 PDF、ALL PPT 8–9 页、`controller/crShiftLeft.ts`、`ibsCrExecutor.ts`、UnifiedExecutor、ExternalCallback、ServerMonitor。
- 最低体量：11,000 有效字符。

### 第 9 章：总结：从单次审查到可持续质量系统

- 核心问题：怎样把本课程的方法迁移到一个新团队，而不是复制一组固定阈值？
- 学习路径：把九章知识重组为“目标—上下文—执行—验证—度量—回流—服务化”的工作流程；用成熟度阶梯选择下一步；最后完成一份迁移计划。
- 主正文类型：课程总结。
- 主教学任务：阶段复习与迁移设计。
- 风格：严谨技术型 + 简洁专业型。
- 必须覆盖：采纳与召回的权衡、质量门的校准、事实/设计/实验的区分、人机职责、落地顺序和未覆盖方向。
- 完成证据：提交 30/60/90 天落地路线，包含基线、最小闭环、失败停止条件和人工审核点。
- 关键图：AICR 工作系统全景；成熟度阶梯；迁移决策树。
- 主要来源：全课已批准来源。
- 最低体量：7,000 有效字符。

## 跨章一致性约束

- “采纳率”首次正式定义在第 3 章，后续如使用旧版文档口径必须明确标记。
- “召回率”必须说明是否包含人工 CR、测试 Bug，以及 Bug 归属过滤。
- “高风险”以代码当前 `score >= 4` 的统计条件为实现事实，设计材料中的 P0/P1/P2 另作映射说明。
- “AI 偷懒”作为可观察的长任务失效简称，不人格化解释模型动机。
- 阈值只用于基线与实验，不包装成行业普适值。
- 本地和远端共用审查 Harness，但触发、工作区、任务生命周期、并发与回调不同。

## 建议批次

- 批次 01–03：背景、架构、指标。
- 批次 04–06：采纳率问题/改造、召回率问题。
- 批次 07–09：召回率改造、远端服务、总结。

每批次完成 Markdown、HTML、Humanizer、确定性检查、语义审校和浏览器体验验证后再交付。
