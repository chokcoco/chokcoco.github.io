# Course Generation Spec：如何实现高质量的 AI Code Review

状态：生成规格已确认 v0.3；当前批次为第 4–5 章。

## 课程形态

- 课程名：《如何实现高质量的 AI Code Review》
- 课程 slug：`high-quality-ai-code-review`
- 章节数：9 章
- 语言：中文；首次出现的关键英文术语保留原文。
- 基础讲师风格：严谨技术型。
- 局部修饰：第 1、9 章使用简洁专业型；架构与诊断章节使用案例分析型；项目步骤使用实践带练型。
- 事实源：Markdown 正文；静态 HTML 由构建脚本生成，不直接手改生成产物。
- 发布状态：默认人工审核，不自动发布。

## 章节顺序

指标章确定放在第 3 章：

1. 为什么今天需要 AI Code Review
2. 从 Prompt 到 Harness：整体架构与核心思路
3. 先建立度量：采纳率、召回率与 F1-Score
4. 采纳率低：问题为什么“看起来对，却没人采用”
5. 提升采纳率：规则、上下文、复核与负向记忆
6. 召回率低：大型任务中 AI 为什么会“偷懒”
7. 提升召回率：分批、状态机、质检与数据回流
8. 远端 AI Code Review 云服务
9. 总结：从单次审查到可持续质量系统

调整理由：第 4–7 章会频繁讨论优化是否有效。先定义统计对象、分母和偏差，后续才能准确解释每个工程动作改变了什么。

## 篇幅口径

- 第 1、9 章：最低 7,000 有效字符，可以超过，不能低于。
- 第 2–8 章：最低 11,000 有效字符，可以超过，不能低于。
- 全课最低约 91,000 有效字符。

建议的有效字符定义：正文纯文本中的中文字符、拉丁字母、数字和用于表达语义的标点；不计空白、Markdown 标记、URL、代码块、Mermaid 源码、HTML 标签和自动生成导航。表格单元格中的教学文字计入，图注计入。

最低字符数用于防止内容缩水，不代表达到数字即合格。超过最低值的部分仍需通过篇幅有效性检查，不能靠重复、长代码或表格复述堆积。

## 教学设计

- 每章只保留一个核心问题和一条可复述的因果链。
- 代码片段以最小必要范围展示，优先使用当前实现中的真实方法；若需公开发布，则改为脱敏节选或等价伪代码。
- 数值阈值区分“实验基线”“当前默认配置”“数据库热配置后的运行值”。
- 图示用于架构、状态、时序、数据流和指标分母；普通清单不转成图。
- 每章至少有一种学习证据：判断、复算、诊断、图解、设计或运行结果。
- 第 1、9 章可用路线检查、决策题和迁移计划代替代码练习。
- 第 2–8 章建议包含一个与 `qms-server` 对应的“源码观察点”，但不强行粘贴完整文件。

## 项目实践

### A. 现有仓库分析与改造方案

学习者阅读一份脱敏后的 `qms-server` 快照，完成状态机、指标口径、质量门、远端服务的架构审计和改造方案。优点是贴近真实工程，实施成本低；缺点是可运行产物较弱。

### B. 可运行的简化 AICR Harness

提供一个小型代码仓库和伪审查器，学习者实现：Setup、分批、状态机、结果提交、验证、指标计算和异步任务。优点是完成证据强；缺点是需要额外维护实验代码和测试。

采用“A 为思维主线，B 为可选支架”的双层实践。最终项目脱离 `qms-server` 的业务命名，重点是设计判断与机制迁移。代码只在佐证实现、验证指标或讲清状态变化时出现。

## 图形与视觉规范

- 核心技术图优先 Mermaid、SVG 或 HTML/CSS 原生绘制，保证中文标签准确、可访问和可维护。
- 预计全课 18–26 张有信息增量的图，不为追求数量重复画同一架构。
- 每张图必须有正文解释、条件和阅读顺序；图本身不作为证据。
- 使用中性技术视觉，不沿用内部 PPT 的品牌样式。
- GPT-IMAGE-2/imagegen 可用于课程封面、章节概念插画或低文字信息图，不用于承载精确状态名、公式、代码字段或复杂中文架构标签。
- 从内部 PPT/PDF 重绘时删除内部 Logo、人员、域名、生产数据和“Private & Confidential”等标识。

## 交付物

- 9 章 Markdown 事实源。
- 静态 HTML 首页、章节页、目录导航、代码高亮和 Mermaid 渲染。
- 术语表、指标速查表和来源清单。
- 不设置机械的每章练习和标准答案。只在有助于检查理解时加入判断、复算、架构阅读或迁移任务，并弱化学生页面中的验收措辞。
- 可运行的简化 AICR 支架按教学需要提供，不以代码规模作为交付目标。
- `quality/` 下的确定性检查、语义审校、Humanizer 和浏览器体验报告。

## 来源与披露

- workspace 可以记录内部材料名、真实路径和断言映射。
- 课程用于公开售卖。学生页面允许展示真实代码片段、接口名和类名；数据库表名、平台名、组织、域名、人员和生产指标必须脱敏。
- 使用脱敏案例、教学样本和重绘图；公共参考文献放在章末，内部资料不出现在学生页面。
- 公开网络资料只采用可追溯的一手来源，数字和易变化事实记录访问日期。

## 质量流程

确认任务定义后，按 `teach-generator` 执行：

1. 每批生成连续 3–4 章的 Markdown 事实源和目标产物。
2. 对每份中文学生内容执行 `humanizer-zh`，低于 45/50 继续修改。
3. 运行 `node .agents/skills/teach-generator/scripts/validate-course.mjs <course-root>`。
4. 第一阶段通过后读取 `references/course-rubric.md` 执行语义审校。
5. 用浏览器检查导航、长文阅读、代码、Mermaid、响应式与本地链接。
6. 任何正文修改使旧报告失效，从第一阶段重新检查。
7. 整课完成后执行跨章术语、指标口径、来源、重复、披露和构建回归。
