# Mission：如何实现高质量的 AI Code Review

状态：任务定义已确认 v0.3；已授权生成至第 5 章，整课仍需分批人工审核。

## 课程为什么存在

这门课解决的不是“怎样写一段 Code Review 提示词”，而是怎样把 AI Code Review 建成一套能进入真实研发流程、可以度量、可以复盘、可以持续改进的工程系统。

课程以 `qms-server` 中的 CR Shift-Left 实现为主案例。学习者会沿着一条真实的演进路线工作：先看到人工 CR 与简单 AI CR 的局限，再分析 Prompt-only 方案在大型任务中的失效，随后用任务编排、状态机、分批审查、结果质检、记忆回流和远端异步服务逐步解决采纳率与召回率问题。

## 核心受众

- 主线受众：有真实研发经验、会使用 Git 和基本 Prompt 的全栈研发工程师。
- 次级受众：TL、架构师、研发效能和质量平台工程师。
- 不默认学习者熟悉 Merge Request、Agent、MCP、Harness、LCS、F1-Score 或评估集建设；这些概念在首次使用前从零解释。

课程以研发工程师的实际审查任务为主线。TL、架构师和研发效能团队通过架构权衡、指标设计和远端服务章节获得进阶启发，不在每一段正文中并行维护第二套叙事。

## 学习成果

完成课程后，学习者应能：

- 解释人工 CR、规则型静态检查、远端 Diff 审查、本地上下文审查和 Harness 驱动审查之间的差异。
- 画出一条 AI CR 请求从触发、Setup、分批、审查、验证、扩展检查到汇总的状态与数据流。
- 区分影响采纳率与召回率的不同原因，不把“发现更多问题”等同于“审查质量更高”。
- 定义并计算采纳率、召回率、Precision、F1-Score，说明分母口径、样本偏差与不可判定状态的影响。
- 解释为什么大型任务会出现审查衰减，以及 Prompt、简单拆分和 AI 自管理流程为什么不足以单独解决问题。
- 设计批次分配、外部状态机、质量门、重试/熔断、上下文增强和数据回流机制。
- 阅读 `qms-server` 的关键实现，能够把设计文档中的概念映射到 Session、Batch、File、Issue、PendingTask 和 MemoryRule 等代码对象。
- 设计本地 AI CR 与远端 AI CR 服务的边界，并说明 Mirror + Worktree、异步队列、并发控制、结果回调、监控和任务复用分别解决什么问题。
- 为一个新的 AI CR 系统提交可执行的架构方案、指标定义、最小验证实验和演进路线。

## 完成证据与实践形态

建议至少保留四类证据：

- 架构证据：一张包含参与方、状态、数据对象、失败点与可观测点的 AI CR 架构图。
- 指标证据：一个能复算采纳率、召回率和 F1 的小型样本，并解释排除项。
- 诊断证据：针对“采纳率低”或“召回率低”的现象提出可验证假设，选择对应的代码与指标验证。
- 项目证据：课程最终脱离 `qms-server` 的具体业务，学习者完成一份 AI CR Harness 设计；可运行的简化流水线作为进阶支架，代码只在能佐证机制或帮助理解时出现。

## 课程范围

课程重点覆盖：

- CR Shift-Left 的背景与发展阶段。
- Prompt-only 与工程化 Harness 的边界。
- 采纳率：规则、上下文、复核过滤、负向记忆和自动采纳检测。
- 召回率：批次编排、状态机、工作量验证、扩展检查、正向/遗漏记忆和测试集。
- 量化评估：口径、采样、偏差、动态测试集与指标闭环。
- 远端服务：标准化接入、异步执行、Git 工作区、并发、复用、回调与监控。

第一版不展开：

- 训练或微调代码审查模型。
- 用一个统一阈值替代不同仓库的基线校准。
- 把 `qms-server` 中与 CR Shift-Left 无关的业务模块写入课程。
- 将尚未验证的内部规划、实验数值或旧版设计描述为当前代码事实。

## 事实与披露边界

- 课程用于公开售卖，学生内容必须达到可公开披露标准。
- 允许展示 `qms-server` 的真实代码片段、接口名和类名。为了教学清晰，可以删去无关分支、日志和业务字段，但必须标明“简化节选”。
- 数据库不出现真实表名；字段、关系和脱敏后的表结构可以展示。
- 团队、业务线、内部平台、域名、账号和人员信息全部脱敏。
- 演进时间限定为 2025 年 9 月至 2026 年 5 月。
- 真实业务数据不直接披露。课程根据现有材料构造近似的教学样本，并明确标注“教学样本”或“模拟数据”。
- 私有 workspace 可以记录内部材料名、真实路径和实现差异；学生页面不得暴露创作过程或内部来源名。
