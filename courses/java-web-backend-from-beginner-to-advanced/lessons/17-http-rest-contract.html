HTTP 与 REST API 契约

HTTP 请求说明客户端想做什么，HTTP 响应说明服务器实际做了什么；REST API 的设计要让这种约定稳定而清楚。

本章先建立 HTTP 的共同语言，再学习怎样把一个待办事项接口写成可被前端、测试和后端共同理解的契约。JDK 17 主线

前一模块学习的是 Java 语言本身。接下来进入 Web 后端时，最先需要理解的不是 Spring 的注解，而是浏览器、前端程序和后端程序之间究竟怎样交流。本章暂不要求你启动 Java Web 服务；第 18 章把服务运行起来后，你可以直接用本章的 curl 命令观察真实的请求和响应。

一、HTTP 是客户端和服务器交换消息的规则

打开一个网页、在前端点击“保存”、移动端刷新列表，背后都会发生一次或多次网络通信。发起通信的一方通常叫作客户端（client），接收请求并给出结果的一方叫作服务器（server）。浏览器是客户端，Vue 或 React 页面运行后也是客户端；Java Web 程序通常承担服务器的角色。

客户端和服务器不必使用同一种语言。前端可以是 JavaScript，后端可以是 Java；它们能协作，是因为双方都遵守同一套 HTTP 规则。HTTP 规定了一条消息如何表达“我要什么”和“结果是什么”。它不规定待办事项有哪些字段，也不替你写业务规则。

HTTP 是无状态协议：从协议的角度看，一次请求本身不会自动携带“上一条请求已经发生过”的记忆。登录状态、购物车、限流计数等能力需要应用通过 Cookie、令牌、数据库或缓存等方式额外实现。本课程后面讲 JWT、Redis 和 Spring Security 时会回到这个问题。

1.1 HTTP、接口和 REST 分别是什么

这三个词经常一起出现，但含义不同：

名称可以先这样理解例子
HTTP客户端与服务器收发消息的通用协议请求行、请求头、状态码
API程序向其他程序开放的使用约定GET /api/todos/42 返回待办事项
REST一种组织 Web API 的设计风格用资源路径、HTTP 方法和状态码表达意图

因此，“做一个 REST API”并不等于“返回 JSON”。JSON 是常见的数据表示格式；REST 更关心资源如何命名、操作意图如何表达、成功和失败怎样被稳定地表示。

3.1 状态码先表达大类，再表达具体结果

状态码的第一位已经表达了结果的大方向：

范围含义常见的后端处理方向
2xx请求成功得到处理正常显示或继续下一步
3xx需要进一步跳转或使用缓存等机制浏览器或客户端按协议继续处理
4xx客户端请求有问题或没有权限修正输入、登录或提示无权访问
5xx服务器未能完成有效请求记录证据、告警并排查服务端问题

场景推荐状态码含义
查询、修改成功并返回数据200 OK成功，响应体有结果

400 和 500 都代表失败，但责任边界不同。用户提交了空标题，属于可预期的输入错误，应该返回 400 并给出可读的字段提示；数据库暂时无法连接，则是服务端或依赖问题，不能伪装成“用户输入错误”。

4.1 方法不是随意的动词标签

HTTP 对一些方法有重要语义，尤其是“安全”和“幂等”两个概念：

方法常见用途安全（safe）幂等（idempotent）
GET读取资源是是
POST在集合下创建资源或提交处理否不保证
PUT用请求表示完整替换目标资源否是
PATCH部分更新资源否不保证，取决于设计
DELETE删除目标资源否是

“安全”不是指网络安全，而是指该方法在语义上不应改变服务器状态。读取接口不应该因为刷新页面就创建一条记录。

“幂等”指同一个有效请求执行一次和执行多次，目标资源的最终状态应相同。对不存在的待办事项重复执行 DELETE /api/todos/42，具体响应可以由接口约定决定，但成功删除后不应因为重试而再删除其他资源。POST 常用于创建，重复提交可能创建两条记录，所以 HTTP 没有把它规定为幂等。支付、下单等必须避免重复处理的场景，通常还需要额外设计幂等键；这是后端进阶主题，不能只靠换一个方法名解决。

六、REST 是一组共同生效的架构约束

REST 是 Representational State Transfer 的缩写，常译为“表述性状态转移”。这个名称容易让初学者把注意力放到单词翻译上。学习 REST 更有效的方式，是先看它对系统交互提出了哪些约束，以及这些约束解决什么问题。

Roy Fielding 在论文中描述的 REST 不是“路径必须使用复数名词”之类的命名清单，而是一种面向分布式超媒体系统的架构风格。它由客户端—服务器、无状态、可缓存、统一接口、分层系统等约束共同组成；按需代码是可选约束。某个 JSON API 使用了 HTTP 和名词路径，只能说明它借鉴了 REST 的一部分，不能据此推断它满足全部 REST 约束。

6.1 客户端与服务器分离职责

客户端负责交互和展示，服务器负责数据与业务能力。分离并不表示前端不能做校验，也不表示后端不能返回适合页面展示的数据。它要求双方通过公开接口协作，不让客户端依赖服务器的类名、表名或进程内对象。

这种边界允许双方独立演进。后端可以把数据从文件迁移到数据库，只要公开契约没有破坏，客户端无需知道；前端也可以从网页换成移动端，只要仍能构造符合契约的请求。

6.2 无状态要求请求携带理解它所需的信息

REST 的无状态约束针对的是客户端与服务器之间的请求交互，不是说服务器不能保存数据。待办事项当然可以存在数据库里，Redis 也可以保存缓存。关键区别是：服务器不应依赖某个只存在于内存会话中的“上一步对话状态”，才能理解当前请求。

假设服务器要求客户端先调用 /selectProject?id=8，随后调用 /todos 时不再携带项目编号，而是依赖服务器记住“当前项目是 8”。这样的接口会让请求难以独立重放，也会增加多实例间复制会话状态的负担。更清楚的请求是：

项目编号和身份凭据都随请求到达。任意一个有相同数据和配置的服务实例都能理解它。身份凭据可以引用服务器保存的账号与权限数据；“请求携带必要上下文”不等于“把所有用户资料都塞进 Token”。

无状态也有代价。请求可能需要重复传递身份和上下文，服务器每次都要完成必要校验。是否保存短期流程状态、怎样处理大文件上传或多步骤事务，需要结合业务设计，不能把“无状态”理解成禁止一切服务端状态。

6.3 可缓存要求响应说明能否被复用

缓存可以让客户端或中间层复用先前响应，减少重复传输和服务器计算。REST 的可缓存约束要求响应隐式或显式地表明是否可缓存。HTTP 通过 Cache-Control、ETag、Last-Modified 等响应头承载这些信息。

认证接口、包含个人数据的响应不能照搬上面的缓存策略。是否允许共享缓存、是否只能由浏览器私有缓存保存、是否完全禁止存储，都要根据数据敏感性和一致性要求决定。no-cache 的含义是复用前需要向源服务器验证，并不等于“绝不保存”；明确禁止存储使用 no-store。第 32 章会讲应用层 Redis 缓存，它和 HTTP 缓存不是同一层机制。

6.4 统一接口让参与方理解同一种交互语言

统一接口是 REST 的核心约束。它不是要求所有资源拥有完全相同的 JSON 字段，而是要求交互遵守一组通用语义。Fielding 将统一接口进一步拆成四项约束：资源可被标识、通过资源的表示进行操作、消息能够自我描述、超媒体驱动应用状态。

资源可被标识，意味着待办事项 42 有稳定的标识，例如 /api/todos/42。资源本身是业务概念，不等同于数据库中的某一行，也不等同于某次返回的 JSON 字符串。

通过表示操作资源，意味着客户端收发的是资源在某一时刻的表示。同一个待办事项可以用 JSON、HTML 或其他媒体类型表示：

消息能够自我描述，意味着接收方可以从方法、目标地址、状态码、Header、媒体类型和消息体理解怎样处理消息。例如只有一段 JSON 而没有 Content-Type，接收方就缺少对格式的明确说明；错误响应只写 失败，也无法让通用客户端判断是认证、输入还是服务端问题。

6.5 分层系统允许中间层参与处理

客户端发出的请求不一定直接到达 Java 进程。它可能先经过 CDN、反向代理、网关、负载均衡或安全过滤层。分层约束让每个组件只需要理解与相邻层的交互，不必知道完整内部拓扑。

这解释了为什么接口不能把“直接连接某台服务器”当作契约。状态码、缓存头和身份信息要能被中间层正确处理；客户端 IP、协议或 Host 等信息经过代理后如何传递，也需要明确的基础设施约定。分层会增加额外跳数和延迟，收益是可以集中处理路由、缓存、安全策略与流量分配。

6.6 按需代码是可选约束

服务器可以向客户端发送脚本，让客户端获得新的执行能力。Web 页面下载 JavaScript 就符合这种思路。它会降低交互的可见性，因此在 REST 中是可选约束。普通后端 JSON API 设计不需要为了满足这一项额外返回脚本。

7.2 PUT 与 PATCH 要写清更新语义

PUT /api/todos/42 通常表示用请求中的完整表示替换目标资源。若请求没有 priority 字段，服务器是清空它、采用默认值，还是拒绝请求，必须明确。PATCH 用于部分修改，但 HTTP 方法名本身没有告诉客户端补丁文档的格式。

一种简单 JSON 合并式约定可能是：请求中没有出现的字段保持不变，出现且值为 null 的字段表示清空。这里马上产生一个边界：本来就不允许为 null 的字段如何处理？数组是整体替换还是追加？答案必须写进契约，不能由前后端各自猜测。

更新接口还会遇到并发覆盖。客户端 A 和 B 都读取版本 7；A 先保存版本 8，B 随后把旧页面内容保存回去，就可能覆盖 A 的修改。接口可以使用版本字段，或利用 ETag 与 If-Match 做条件更新。条件不满足时返回 412 Precondition Failed，提示客户端重新读取并处理冲突。初学阶段先理解问题和契约，具体持久化控制会在数据与事务章节展开。

八、表示、媒体类型与内容协商

JSON 是 Java Web API 最常见的表示格式，但 HTTP 不把资源限制为 JSON。Content-Type 给出当前消息体的媒体类型，Accept 表示客户端可接受的响应媒体类型，这一过程称为内容协商。

如果接口只支持 JSON，就可以固定返回 Content-Type: application/json。若客户端请求只接受服务器不支持的格式，服务器可以返回 406 Not Acceptable；客户端发送服务器不支持的请求体格式时，可以返回 415 Unsupported Media Type。它们比统一返回 400 更准确地表达失败位置。

字段也是表示契约。createdAt 是带时区的时间点还是本地时间？金额是数值还是带币种的对象？枚举增加新值时旧客户端怎样处理？这些问题不能由 JSON 语法回答。第 22 章会具体讲 DTO 与 Jackson；本章要先形成判断：类型、可空性、格式、取值范围和兼容策略都属于表示契约。

type 标识问题类型，最好能指向该问题的说明；instance 标识这一次具体问题。团队仍可以增加字段级错误列表，但扩展字段需要稳定定义。Problem Details 不是强制所有 API 使用的唯一错误格式，也不能代替正确状态码。已有成熟错误契约的系统可以继续使用自己的格式；新系统选择它时，要在客户端和服务端统一媒体类型、字段含义与扩展方式。

十、契约演进先判断变化是否破坏现有客户端

接口上线后，最难处理的不是第一次设计，而是变化。新增可选响应字段通常不会破坏能忽略未知字段的客户端；删除字段、把字符串改成对象、把可选字段改为必填、改变状态码含义，都可能让旧客户端失败。

版本号不是自动兼容工具。即使使用 /api/v1/todos，也要定义 v1 内允许哪些兼容变化、旧版本支持多久、客户端如何迁移。只有出现无法兼容的语义变化时再建立新版本，通常比每次加字段都升级版本更容易维护。

契约评审至少要同时看正常路径和失败路径：请求字段、响应字段、方法与路径、成功状态码、错误状态码、错误体、分页、认证、幂等与兼容性。OpenAPI 等描述工具可以帮助保存这些信息，但工具生成的文档仍然可能语义含糊；表格齐全不等于设计已经完成。

十三、常见误区

13.1 把所有失败都返回 200

有些接口把“业务失败”装进 { "success": false }，但仍返回 200。这样会丢掉 HTTP 本身提供的错误语义，网关、监控、客户端通用逻辑也更难正确处理。可预期的参数错误返回合适的 4xx，服务端故障返回 5xx，再在响应体中提供稳定错误信息。

13.2 用 GET 触发删除或写入

搜索引擎、浏览器预取、缓存和用户刷新都可能重复发起 GET。如果 GET /deleteTodo?id=42 会删除数据，就违背了读取方法的安全语义，也会制造难以复现的数据问题。

13.3 只看路径，不看方法

GET /api/todos/42 和 DELETE /api/todos/42 的路径相同，含义却完全不同。阅读接口文档、排查日志或写路由时，要把“方法 + 路径”作为一个整体。

13.4 把内部异常原样返回

NullPointerException、数据库用户名或完整堆栈对最终用户没有帮助，还可能造成信息泄露。对外返回契约化错误，对内保留足够的日志和异常链，二者服务的对象不同。

13.5 把无状态理解成服务器不能保存数据

无状态约束说的是每个请求需要携带理解它所需的信息，不是禁止数据库、缓存和账号状态。把业务数据持久化与依赖内存会话理解为一件事，会让 REST 设计走向错误的极端。

13.6 认为名词路径加 JSON 就是完整 REST

资源路径和 JSON 只是可见表面。消息是否自描述、方法语义是否正确、响应能否缓存、客户端是否依赖服务端会话、是否利用超媒体，都属于 REST 约束。工程中可以只采用其中一部分，但文档应准确说明，而不是用“RESTful”替代具体设计。

十四、本章小结

HTTP 是客户端与服务器交换消息的规则。一次请求由方法、路径、查询参数、请求头和可选请求体组成；一次响应由状态行、响应头和可选响应体组成。REST API 用资源路径和 HTTP 方法表达意图，用状态码和结构化响应表达结果。先把这些约定设计清楚，后续 Spring Controller、参数校验、认证和异常处理才有明确的落点。

  5. 为什么错误响应需要稳定的 code，而不应只依赖 message？
  6. REST 的无状态约束为什么不等于服务器不能使用数据库？
  7. 资源与资源的 JSON 表示有什么区别？
  8. no-cache 与 no-store 的含义有什么不同？
  9. 条件更新中的 If-Match 可以避免哪类覆盖问题？

参考答案：Content-Type 说明发送内容的格式，Accept 说明期望的响应格式；GET 在语义上应当安全；201 明确表示新资源已创建；400 通常是请求不符合约定，500 是服务器未预期地失败；message 可能因语言或文案变化，code 更适合程序判断。无状态要求单次请求包含必要上下文，数据库保存的是资源与业务状态；资源是被识别的业务概念，JSON 只是它的一种表示；no-cache 要求复用前验证，no-store 禁止存储；If-Match 可以检测客户端基于旧版本提交的更新，避免静默覆盖较新的修改。
