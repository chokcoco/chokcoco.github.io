# 第 17 章　HTTP 与 REST API 契约

> 学习提示：本章先建立 HTTP 的共同语言，再学习怎样把一个待办事项接口写成可被前端、测试和后端共同理解的契约。
> 一句话总结：HTTP 请求说明客户端想做什么，HTTP 响应说明服务器实际做了什么；REST API 的设计要让这种约定稳定而清楚。

前一模块学习的是 Java 语言本身。接下来进入 Web 后端时，最先需要理解的不是 Spring 的注解，而是浏览器、前端程序和后端程序之间究竟怎样交流。本章暂不要求你启动 Java Web 服务；第 18 章把服务运行起来后，你可以直接用本章的 `curl` 命令观察真实的请求和响应。

## 一、HTTP 是客户端和服务器交换消息的规则

打开一个网页、在前端点击“保存”、移动端刷新列表，背后都会发生一次或多次网络通信。发起通信的一方通常叫作**客户端**（client），接收请求并给出结果的一方叫作**服务器**（server）。浏览器是客户端，Vue 或 React 页面运行后也是客户端；Java Web 程序通常承担服务器的角色。

客户端和服务器不必使用同一种语言。前端可以是 JavaScript，后端可以是 Java；它们能协作，是因为双方都遵守同一套 [[HTTP]] 规则。HTTP 规定了一条消息如何表达“我要什么”和“结果是什么”。它不规定待办事项有哪些字段，也不替你写业务规则。

可以先把一次交流想成下面的过程：

```text
客户端                                      Java Web 服务
  |                                             |
  |  GET /api/todos/42                          |
  |-------------------------------------------->|
  |                                             | 查找编号为 42 的待办事项
  |  200 OK + JSON 数据                          |
  |<--------------------------------------------|
```

这里的 `GET /api/todos/42` 是请求的一部分，`200 OK` 和 JSON 数据是响应的一部分。客户端只应根据响应中公开的契约判断结果，不能猜测服务器内部使用了哪张表、哪个 Java 类或哪种缓存。

HTTP 是无状态协议：从协议的角度看，一次请求本身不会自动携带“上一条请求已经发生过”的记忆。登录状态、购物车、限流计数等能力需要应用通过 Cookie、令牌、数据库或缓存等方式额外实现。本课程后面讲 JWT、Redis 和 Spring Security 时会回到这个问题。

### 1.1 HTTP、接口和 REST 分别是什么

这三个词经常一起出现，但含义不同：

| 名称 | 可以先这样理解 | 例子 |
| --- | --- | --- |
| HTTP | 客户端与服务器收发消息的通用协议 | 请求行、请求头、状态码 |
| API | 程序向其他程序开放的使用约定 | `GET /api/todos/42` 返回待办事项 |
| REST | 一种组织 Web API 的设计风格 | 用资源路径、HTTP 方法和状态码表达意图 |

因此，“做一个 REST API”并不等于“返回 JSON”。JSON 是常见的数据表示格式；REST 更关心资源如何命名、操作意图如何表达、成功和失败怎样被稳定地表示。

## 二、读懂一条 HTTP 请求

一条请求由请求行、请求头和可选的请求体组成。先看一个查询待办事项的简化请求：

```http
GET /api/todos/42?include=tags HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer <token>
```

不要急着背格式。逐项理解它表达的意思：

| 部分 | 示例 | 作用 |
| --- | --- | --- |
| 方法（method） | `GET` | 表达本次希望进行的操作类型 |
| 路径（path） | `/api/todos/42` | 指向一个资源或资源集合 |
| 查询参数（query） | `include=tags` | 补充筛选、分页或展示选项 |
| 协议版本 | `HTTP/1.1` | 表示所用 HTTP 版本 |
| 请求头（header） | `Accept: application/json` | 提供格式、身份、缓存等附加信息 |
| 请求体（body） | 此例没有 | 传递创建或修改时的数据 |

`/api/todos/42?include=tags` 可以拆成两部分：`/api/todos/42` 是路径，`?` 后的 `include=tags` 是查询参数。路径通常回答“操作哪个资源”，查询参数通常回答“以什么条件或形式查询”。例如：

```text
/api/todos?page=1&pageSize=20&status=OPEN
```

上面的地址仍然是查询待办事项集合；`page`、`pageSize` 和 `status` 只是补充了分页和过滤条件。查询参数不是 Java 变量，它们最初是 URL 中的文本；后端必须检查它们是否能转换为期望的类型、是否位于允许范围内。

### 2.1 请求体什么时候出现

创建或修改资源时，客户端常用请求体携带 JSON。下面是一条创建待办事项的简化请求：

```http
POST /api/todos HTTP/1.1
Host: api.example.com
Content-Type: application/json
Accept: application/json

{
  "title": "阅读 HTTP 规范",
  "priority": "HIGH"
}
```

空行很重要：它把请求头和请求体分开。`Content-Type: application/json` 是在告诉服务器“请求体按 JSON 格式组织”。`Accept: application/json` 表示客户端希望服务器优先返回 JSON。二者一个描述**我发送的内容**，一个描述**我希望接收的内容**，不要混淆。

JSON 中的对象、数组、字符串、数值、布尔值和 `null` 与 Java 对象并不天然相同。第 19 章会讲 Spring 如何把 JSON 绑定到 Java 对象；本章只需要记住：字段名、字段类型和是否必填，都是接口契约的一部分。

## 三、响应把处理结果交给客户端

服务器收到请求后，会返回一条 HTTP 响应。一个成功查询的简化响应如下：

```http
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: no-store

{
  "id": 42,
  "title": "阅读 HTTP 规范",
  "priority": "HIGH",
  "completed": false
}
```

响应同样由三部分组成：

| 部分 | 示例 | 作用 |
| --- | --- | --- |
| 状态行 | `HTTP/1.1 200 OK` | 协议版本、状态码和简短原因短语 |
| 响应头 | `Content-Type: application/json` | 说明响应格式、缓存策略等元数据 |
| 响应体 | JSON 对象 | 返回给客户端的业务数据或错误信息 |

最常被程序读取的是三位数的[[状态码]]，例如 `200`。`OK` 是帮助人阅读的原因短语；前端业务代码不应依赖原因短语判断逻辑。响应体也不一定总存在，例如删除成功时常使用 `204 No Content`，它的含义就是成功但没有响应体。

### 3.1 状态码先表达大类，再表达具体结果

状态码的第一位已经表达了结果的大方向：

| 范围 | 含义 | 常见的后端处理方向 |
| --- | --- | --- |
| `2xx` | 请求成功得到处理 | 正常显示或继续下一步 |
| `3xx` | 需要进一步跳转或使用缓存等机制 | 浏览器或客户端按协议继续处理 |
| `4xx` | 客户端请求有问题或没有权限 | 修正输入、登录或提示无权访问 |
| `5xx` | 服务器未能完成有效请求 | 记录证据、告警并排查服务端问题 |

后端接口中常见的选择如下：

| 场景 | 推荐状态码 | 含义 |
| --- | --- | --- |
| 查询、修改成功并返回数据 | `200 OK` | 成功，响应体有结果 |
| 成功创建一个新资源 | `201 Created` | 已创建；可用 `Location` 响应头给出新资源地址 |
| 成功删除，且无需返回内容 | `204 No Content` | 成功，响应体应为空 |
| 字段缺失、格式错误、分页参数不合法 | `400 Bad Request` | 请求不能按约定被处理 |
| 未登录或身份凭证无效 | `401 Unauthorized` | 需要有效身份认证；具体认证流程后面再讲 |
| 已登录但无权访问 | `403 Forbidden` | 身份已知，但权限不足 |
| 资源不存在 | `404 Not Found` | 指定资源没有找到 |
| 状态冲突，例如重复提交 | `409 Conflict` | 请求与当前资源状态冲突 |
| 未预期的服务端错误 | `500 Internal Server Error` | 服务端失败，不应把堆栈直接返回给用户 |

`400` 和 `500` 都代表失败，但责任边界不同。用户提交了空标题，属于可预期的输入错误，应该返回 `400` 并给出可读的字段提示；数据库暂时无法连接，则是服务端或依赖问题，不能伪装成“用户输入错误”。

## 四、用资源、方法和路径表达意图

REST API 通常把业务对象称为[[资源]]。待办事项、订单、用户都可以是资源。资源集合和单个资源常用名词路径表示：

```text
GET    /api/todos          查询待办事项集合
POST   /api/todos          创建一个待办事项
GET    /api/todos/42       查询编号为 42 的待办事项
PUT    /api/todos/42       用完整表示替换编号为 42 的待办事项
PATCH  /api/todos/42       修改编号为 42 的部分字段
DELETE /api/todos/42       删除编号为 42 的待办事项
```

路径主要使用名词，操作意图主要由 HTTP 方法表达。`/api/todos/42` 本身只是“哪个待办事项”；到底是读取、更新还是删除，要由 `GET`、`PATCH`、`DELETE` 等方法一起判断。相比 `/getTodo?id=42` 或 `/deleteTodo?id=42`，这种组合更容易扩展，也更符合 HTTP 的通用语义。

### 4.1 方法不是随意的动词标签

HTTP 对一些方法有重要语义，尤其是“安全”和“幂等”两个概念：

| 方法 | 常见用途 | 安全（safe） | 幂等（idempotent） |
| --- | --- | --- | --- |
| `GET` | 读取资源 | 是 | 是 |
| `POST` | 在集合下创建资源或提交处理 | 否 | 不保证 |
| `PUT` | 用请求表示完整替换目标资源 | 否 | 是 |
| `PATCH` | 部分更新资源 | 否 | 不保证，取决于设计 |
| `DELETE` | 删除目标资源 | 否 | 是 |

“安全”不是指网络安全，而是指该方法在语义上不应改变服务器状态。读取接口不应该因为刷新页面就创建一条记录。

“幂等”指同一个有效请求执行一次和执行多次，目标资源的最终状态应相同。对不存在的待办事项重复执行 `DELETE /api/todos/42`，具体响应可以由接口约定决定，但成功删除后不应因为重试而再删除其他资源。`POST` 常用于创建，重复提交可能创建两条记录，所以 HTTP 没有把它规定为幂等。支付、下单等必须避免重复处理的场景，通常还需要额外设计幂等键；这是后端进阶主题，不能只靠换一个方法名解决。

### 4.2 什么时候可以出现动作路径

并非所有业务都能自然描述成“增删改查”。例如“取消订单”可能包含库存、退款和状态转换。此时可以把动作设计为资源的子路径，例如：

```text
POST /api/orders/1001/cancellations
```

这里创建的是“取消申请”这一资源，而不是把所有动词都塞进路径。这个例子用来说明思路；在真正设计前，还要明确谁能取消、重复取消怎样处理、取消成功返回什么状态码。

## 五、把成功和失败都写成稳定的契约

接口契约不只有成功 JSON。调用方还需要知道：字段缺失时返回什么、找不到资源时怎样区分、服务端异常时是否可以安全展示错误信息。

下面是一个校验失败的响应示例：

```http
HTTP/1.1 400 Bad Request
Content-Type: application/json

{
  "timestamp": "2026-07-14T10:30:00Z",
  "status": 400,
  "code": "TODO_TITLE_REQUIRED",
  "message": "title 不能为空",
  "path": "/api/todos"
}
```

这个 JSON 不是唯一正确格式，但它展示了一个有用原则：机器可判断的信息和给人看的信息分开保存。

| 字段 | 面向谁 | 作用 |
| --- | --- | --- |
| `status` | 客户端、日志、监控 | 与 HTTP 状态码对应，便于统一处理 |
| `code` | 前端程序、测试 | 稳定的业务错误标识，例如展示特定提示或埋点 |
| `message` | 使用者、开发者 | 可以阅读的错误说明 |
| `path`、`timestamp` | 排查人员 | 帮助定位发生位置和时间 |

不要把 Java 异常堆栈、SQL 语句、数据库地址或令牌放进响应体。它们可能暴露内部实现和敏感信息，也不能让客户端据此建立稳定逻辑。第 20 章会把参数校验和全局异常处理落到 Spring Boot 代码中；本章先确定它们要对外呈现的行为。

## 六、REST 是一组共同生效的架构约束

[[REST]] 是 Representational State Transfer 的缩写，常译为“表述性状态转移”。这个名称容易让初学者把注意力放到单词翻译上。学习 REST 更有效的方式，是先看它对系统交互提出了哪些约束，以及这些约束解决什么问题。

Roy Fielding 在论文中描述的 REST 不是“路径必须使用复数名词”之类的命名清单，而是一种面向分布式超媒体系统的架构风格。它由客户端—服务器、无状态、可缓存、统一接口、分层系统等约束共同组成；按需代码是可选约束。某个 JSON API 使用了 HTTP 和名词路径，只能说明它借鉴了 REST 的一部分，不能据此推断它满足全部 REST 约束。

### 6.1 客户端与服务器分离职责

客户端负责交互和展示，服务器负责数据与业务能力。分离并不表示前端不能做校验，也不表示后端不能返回适合页面展示的数据。它要求双方通过公开接口协作，不让客户端依赖服务器的类名、表名或进程内对象。

这种边界允许双方独立演进。后端可以把数据从文件迁移到数据库，只要公开契约没有破坏，客户端无需知道；前端也可以从网页换成移动端，只要仍能构造符合契约的请求。

### 6.2 无状态要求请求携带理解它所需的信息

REST 的无状态约束针对的是客户端与服务器之间的请求交互，不是说服务器不能保存数据。待办事项当然可以存在数据库里，Redis 也可以保存缓存。关键区别是：服务器不应依赖某个只存在于内存会话中的“上一步对话状态”，才能理解当前请求。

假设服务器要求客户端先调用 `/selectProject?id=8`，随后调用 `/todos` 时不再携带项目编号，而是依赖服务器记住“当前项目是 8”。这样的接口会让请求难以独立重放，也会增加多实例间复制会话状态的负担。更清楚的请求是：

```http
GET /api/projects/8/todos HTTP/1.1
Host: api.example.com
Accept: application/json
Authorization: Bearer <token>
```

项目编号和身份凭据都随请求到达。任意一个有相同数据和配置的服务实例都能理解它。身份凭据可以引用服务器保存的账号与权限数据；“请求携带必要上下文”不等于“把所有用户资料都塞进 Token”。

无状态也有代价。请求可能需要重复传递身份和上下文，服务器每次都要完成必要校验。是否保存短期流程状态、怎样处理大文件上传或多步骤事务，需要结合业务设计，不能把“无状态”理解成禁止一切服务端状态。

### 6.3 可缓存要求响应说明能否被复用

缓存可以让客户端或中间层复用先前响应，减少重复传输和服务器计算。REST 的可缓存约束要求响应隐式或显式地表明是否可缓存。HTTP 通过 `Cache-Control`、`ETag`、`Last-Modified` 等响应头承载这些信息。

例如，一个公开且短时间内变化不频繁的分类列表可以返回：

```http
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=60
ETag: "category-v12"

[{"id":1,"name":"工作"},{"id":2,"name":"生活"}]
```

`max-age=60` 表示响应在规定的 60 秒新鲜期内可以被复用。`ETag` 是当前表示的验证标识；过期后客户端可以带上 `If-None-Match: "category-v12"` 请求服务器重新验证。如果表示没有改变，服务器可以返回 `304 Not Modified`，省去再次发送响应体。

认证接口、包含个人数据的响应不能照搬上面的缓存策略。是否允许共享缓存、是否只能由浏览器私有缓存保存、是否完全禁止存储，都要根据数据敏感性和一致性要求决定。`no-cache` 的含义是复用前需要向源服务器验证，并不等于“绝不保存”；明确禁止存储使用 `no-store`。第 32 章会讲应用层 Redis 缓存，它和 HTTP 缓存不是同一层机制。

### 6.4 统一接口让参与方理解同一种交互语言

统一接口是 REST 的核心约束。它不是要求所有资源拥有完全相同的 JSON 字段，而是要求交互遵守一组通用语义。Fielding 将统一接口进一步拆成四项约束：资源可被标识、通过资源的表示进行操作、消息能够自我描述、超媒体驱动应用状态。

资源可被标识，意味着待办事项 `42` 有稳定的标识，例如 `/api/todos/42`。资源本身是业务概念，不等同于数据库中的某一行，也不等同于某次返回的 JSON 字符串。

通过表示操作资源，意味着客户端收发的是资源在某一时刻的[[表示]]。同一个待办事项可以用 JSON、HTML 或其他媒体类型表示：

```json
{"id":42,"title":"阅读 HTTP 规范","completed":false}
```

上面的 JSON 是资源的表示，不是资源本体。服务器可以在不改变资源身份的情况下增加一个计算字段，也可以根据 `Accept` 返回另一种媒体类型。客户端提交一个更新表示后，服务器依据业务规则修改资源状态；它不是把客户端传来的文本逐字保存就算完成。

消息能够自我描述，意味着接收方可以从方法、目标地址、状态码、Header、媒体类型和消息体理解怎样处理消息。例如只有一段 JSON 而没有 `Content-Type`，接收方就缺少对格式的明确说明；错误响应只写 `失败`，也无法让通用客户端判断是认证、输入还是服务端问题。

超媒体驱动应用状态常被简称为 HATEOAS。服务器在表示中提供可继续执行的链接或操作，客户端根据当前响应发现下一步，而不是把所有流程地址写死。例如：

```json
{
  "id": 42,
  "title": "阅读 HTTP 规范",
  "completed": false,
  "links": [
    {"rel": "self", "href": "/api/todos/42"},
    {"rel": "complete", "href": "/api/todos/42/completion"}
  ]
}
```

很多企业 JSON API 没有完整采用超媒体约束，却仍被团队称为“REST API”。阅读这类文档时，应具体说明它采用了哪些 HTTP 与资源建模原则，不必为了术语争论掩盖契约本身的问题。本课程后续实现的是常见的资源型 HTTP API，同时保留对完整 REST 约束的准确理解。

### 6.5 分层系统允许中间层参与处理

客户端发出的请求不一定直接到达 Java 进程。它可能先经过 CDN、反向代理、网关、负载均衡或安全过滤层。分层约束让每个组件只需要理解与相邻层的交互，不必知道完整内部拓扑。

这解释了为什么接口不能把“直接连接某台服务器”当作契约。状态码、缓存头和身份信息要能被中间层正确处理；客户端 IP、协议或 Host 等信息经过代理后如何传递，也需要明确的基础设施约定。分层会增加额外跳数和延迟，收益是可以集中处理路由、缓存、安全策略与流量分配。

### 6.6 按需代码是可选约束

服务器可以向客户端发送脚本，让客户端获得新的执行能力。Web 页面下载 JavaScript 就符合这种思路。它会降低交互的可见性，因此在 REST 中是可选约束。普通后端 JSON API 设计不需要为了满足这一项额外返回脚本。

## 七、资源建模决定 API 是否容易长期维护

理解 REST 约束后，再回到具体接口。资源建模的第一步不是写 URL，而是列出业务中需要被识别、查询和改变的对象及其关系。

待办应用可能有项目、待办事项、标签和完成记录。下面几条地址表达了不同关系：

```text
/api/projects/8                 编号为 8 的项目
/api/projects/8/todos           项目 8 下的待办事项集合
/api/todos/42                   编号为 42 的待办事项
/api/todos/42/tags              待办事项 42 关联的标签集合
/api/todos/42/completion        待办事项 42 的完成状态或完成操作资源
```

嵌套路径适合表达明确的归属关系，但层级不宜无限加深。`/companies/1/departments/2/teams/3/users/4/todos/5` 很难阅读，资源移动归属时也容易使地址失效。若待办事项 `5` 在系统内有独立身份，查询它可以直接使用 `/api/todos/5`；项目范围则放到集合查询或权限校验中。

### 7.1 集合查询需要约定过滤、排序和分页

集合接口不能只写一句“支持分页”。调用方至少需要知道参数名称、默认值、上下界、排序规则和结果元数据。例如：

```http
GET /api/todos?projectId=8&completed=false&page=1&pageSize=20&sort=createdAt,desc HTTP/1.1
```

一个契约可以规定页码从 1 开始，`pageSize` 默认 20、最大 100，未提供排序时按 `createdAt` 降序。另一套系统可以从 0 开始；问题不在于选了哪个数字，而在于文档、实现和前端是否一致。

页码分页容易理解，但数据持续插入时可能出现重复或遗漏。数据量大、顺序稳定要求高时，可以采用游标分页：响应返回 `nextCursor`，客户端把它带到下一次查询。游标如何生成属于服务端内部实现，契约需要说明游标是否有有效期、排序是否固定，以及没有下一页时返回什么。

### 7.2 PUT 与 PATCH 要写清更新语义

`PUT /api/todos/42` 通常表示用请求中的完整表示替换目标资源。若请求没有 `priority` 字段，服务器是清空它、采用默认值，还是拒绝请求，必须明确。`PATCH` 用于部分修改，但 HTTP 方法名本身没有告诉客户端补丁文档的格式。

一种简单 JSON 合并式约定可能是：请求中没有出现的字段保持不变，出现且值为 `null` 的字段表示清空。这里马上产生一个边界：本来就不允许为 `null` 的字段如何处理？数组是整体替换还是追加？答案必须写进契约，不能由前后端各自猜测。

更新接口还会遇到并发覆盖。客户端 A 和 B 都读取版本 7；A 先保存版本 8，B 随后把旧页面内容保存回去，就可能覆盖 A 的修改。接口可以使用版本字段，或利用 `ETag` 与 `If-Match` 做条件更新。条件不满足时返回 `412 Precondition Failed`，提示客户端重新读取并处理冲突。初学阶段先理解问题和契约，具体持久化控制会在数据与事务章节展开。

## 八、表示、媒体类型与内容协商

JSON 是 Java Web API 最常见的表示格式，但 HTTP 不把资源限制为 JSON。`Content-Type` 给出当前消息体的媒体类型，`Accept` 表示客户端可接受的响应媒体类型，这一过程称为[[内容协商]]。

```http
GET /api/todos/42 HTTP/1.1
Host: api.example.com
Accept: application/json
Accept-Language: zh-CN
```

如果接口只支持 JSON，就可以固定返回 `Content-Type: application/json`。若客户端请求只接受服务器不支持的格式，服务器可以返回 `406 Not Acceptable`；客户端发送服务器不支持的请求体格式时，可以返回 `415 Unsupported Media Type`。它们比统一返回 `400` 更准确地表达失败位置。

字段也是表示契约。`createdAt` 是带时区的时间点还是本地时间？金额是数值还是带币种的对象？枚举增加新值时旧客户端怎样处理？这些问题不能由 JSON 语法回答。第 22 章会具体讲 DTO 与 Jackson；本章要先形成判断：类型、可空性、格式、取值范围和兼容策略都属于表示契约。

## 九、错误响应也可以采用标准问题详情格式

前面的自定义错误结构适合课程逐字段理解。IETF RFC 9457 还定义了 HTTP API 的 Problem Details 格式，JSON 媒体类型为 `application/problem+json`。它常见的成员包括 `type`、`title`、`status`、`detail` 和 `instance`：

```http
HTTP/1.1 400 Bad Request
Content-Type: application/problem+json

{
  "type": "https://api.example.com/problems/validation-error",
  "title": "请求参数校验失败",
  "status": 400,
  "detail": "title 不能为空",
  "instance": "/api/todos"
}
```

`type` 标识问题类型，最好能指向该问题的说明；`instance` 标识这一次具体问题。团队仍可以增加字段级错误列表，但扩展字段需要稳定定义。Problem Details 不是强制所有 API 使用的唯一错误格式，也不能代替正确状态码。已有成熟错误契约的系统可以继续使用自己的格式；新系统选择它时，要在客户端和服务端统一媒体类型、字段含义与扩展方式。

## 十、契约演进先判断变化是否破坏现有客户端

接口上线后，最难处理的不是第一次设计，而是变化。新增可选响应字段通常不会破坏能忽略未知字段的客户端；删除字段、把字符串改成对象、把可选字段改为必填、改变状态码含义，都可能让旧客户端失败。

版本号不是自动兼容工具。即使使用 `/api/v1/todos`，也要定义 v1 内允许哪些兼容变化、旧版本支持多久、客户端如何迁移。只有出现无法兼容的语义变化时再建立新版本，通常比每次加字段都升级版本更容易维护。

契约评审至少要同时看正常路径和失败路径：请求字段、响应字段、方法与路径、成功状态码、错误状态码、错误体、分页、认证、幂等与兼容性。OpenAPI 等描述工具可以帮助保存这些信息，但工具生成的文档仍然可能语义含糊；表格齐全不等于设计已经完成。

例如，列表接口可以返回：

```json
{
  "items": [
    {"id": 42, "title": "阅读 HTTP 规范", "completed": false}
  ],
  "page": 1,
  "pageSize": 20,
  "total": 1
}
```

把数组包进 `items` 并附带分页信息，往往比裸数组更适合长期演进。另一方面，如果当前业务没有分页需求，也不必为了“看起来 REST”凭空加入许多字段。契约要以调用方真实需要的信息为依据，并为可预见的变化留下清楚的演进空间。

这个结构把结果和分页元数据分开。是否返回 `total` 要看查询成本和页面需求；大数据量场景计算总数可能昂贵，游标分页也未必存在准确总数。不要因为模板里有 `total` 就让每次查询承担无用成本。

## 十一、用 curl 直接观察 HTTP 契约

[[curl]] 是一个命令行 HTTP 客户端。它不会替你理解接口，但能让你跳过页面直接查看状态行、响应头和响应体。第 18 章运行起本地服务后，再执行下面的命令；在此之前，`localhost:8080` 没有对应服务时会得到连接失败，这正说明客户端没有连到服务器。

查询一个待办事项：

```bash
curl -i http://localhost:8080/api/todos/42
# -i 表示把响应头也打印到控制台。
# 服务存在且资源存在时，控制台会先显示类似 HTTP/1.1 200 的状态行，随后显示 JSON 响应体。
```

创建一个待办事项。macOS 的终端和 Windows PowerShell 都可以使用单引号包住这段 JSON；PowerShell 中请使用 `curl.exe`，以明确调用 curl 程序而不是同名别名：

```bash
# macOS 终端
curl -i -X POST http://localhost:8080/api/todos \
  -H 'Content-Type: application/json' \
  --data '{"title":"阅读 HTTP 规范","priority":"HIGH"}'

# Windows PowerShell
curl.exe -i -X POST http://localhost:8080/api/todos `
  -H 'Content-Type: application/json' `
  --data '{"title":"阅读 HTTP 规范","priority":"HIGH"}'
```

`-X POST` 指定方法，`-H` 添加请求头，`--data` 发送请求体。命令返回 `201`、`400` 还是 `500` 都是可观察的事实；先记录状态码和响应体，再判断是契约、输入还是服务端实现的问题。

还可以使用 `-v` 查看请求行、请求头、连接过程和响应信息。它会输出更多调试内容，其中可能包含 `Authorization`、Cookie 等敏感值，保存日志或提交截图前要脱敏：

```bash
curl -v http://localhost:8080/api/todos/42
# 控制台中以 > 开头的行主要是发出的请求，以 < 开头的行主要是收到的响应。
# 调试输出默认写到标准错误，响应体仍写到标准输出。
```

## 十二、动手设计一个最小待办事项契约

先不写 Java 代码，为下面三个需求写出 API 契约：

1. 分页查询未完成待办事项，每页 20 条；
2. 创建待办事项，`title` 不能为空；
3. 查询不存在的编号时给出明确结果。

可以先用下表完成设计：

| 需求 | 方法和地址 | 成功结果 | 失败结果 |
| --- | --- | --- | --- |
| 查询未完成列表 | `GET /api/todos?completed=false&page=1&pageSize=20` | `200` 与包含 `items` 的 JSON | 分页参数不合法时 `400` |
| 创建待办事项 | `POST /api/todos` | `201` 与新资源表示 | `title` 为空时 `400` 和稳定错误码 |
| 查询单项 | `GET /api/todos/{id}` | `200` 与待办 JSON | 不存在时 `404` |

再补完下面的契约检查项：

1. 列表中的 `page` 从 0 还是从 1 开始，超出范围怎样处理；
2. 创建成功时是否返回新资源的 `id`，是否提供 `Location`；
3. 重复提交创建请求是否会生成两条数据，是否需要幂等键；
4. 更新时使用 `PUT` 还是 `PATCH`，缺失字段和 `null` 分别表示什么；
5. 错误体采用自定义结构还是 Problem Details，字段级错误放在哪里；
6. 响应是否允许缓存，带个人数据的响应如何限制缓存。

这些问题没有一套适用于所有系统的答案，但不能留给实现偶然决定。完成练习的标准是：另一个没有参与设计的人，只看契约就能构造请求，并能根据状态码与响应体区分成功、输入错误、资源不存在和服务端故障。

## 十三、常见误区

### 13.1 把所有失败都返回 `200`

有些接口把“业务失败”装进 `{ "success": false }`，但仍返回 `200`。这样会丢掉 HTTP 本身提供的错误语义，网关、监控、客户端通用逻辑也更难正确处理。可预期的参数错误返回合适的 `4xx`，服务端故障返回 `5xx`，再在响应体中提供稳定错误信息。

### 13.2 用 `GET` 触发删除或写入

搜索引擎、浏览器预取、缓存和用户刷新都可能重复发起 `GET`。如果 `GET /deleteTodo?id=42` 会删除数据，就违背了读取方法的安全语义，也会制造难以复现的数据问题。

### 13.3 只看路径，不看方法

`GET /api/todos/42` 和 `DELETE /api/todos/42` 的路径相同，含义却完全不同。阅读接口文档、排查日志或写路由时，要把“方法 + 路径”作为一个整体。

### 13.4 把内部异常原样返回

`NullPointerException`、数据库用户名或完整堆栈对最终用户没有帮助，还可能造成信息泄露。对外返回契约化错误，对内保留足够的日志和异常链，二者服务的对象不同。

### 13.5 把无状态理解成服务器不能保存数据

无状态约束说的是每个请求需要携带理解它所需的信息，不是禁止数据库、缓存和账号状态。把业务数据持久化与依赖内存会话理解为一件事，会让 REST 设计走向错误的极端。

### 13.6 认为名词路径加 JSON 就是完整 REST

资源路径和 JSON 只是可见表面。消息是否自描述、方法语义是否正确、响应能否缓存、客户端是否依赖服务端会话、是否利用超媒体，都属于 REST 约束。工程中可以只采用其中一部分，但文档应准确说明，而不是用“RESTful”替代具体设计。

## 十四、本章小结

HTTP 是客户端与服务器交换消息的规则。一次请求由方法、路径、查询参数、请求头和可选请求体组成；一次响应由状态行、响应头和可选响应体组成。REST API 用资源路径和 HTTP 方法表达意图，用状态码和结构化响应表达结果。先把这些约定设计清楚，后续 Spring Controller、参数校验、认证和异常处理才有明确的落点。

## 十五、快速自测

1. `Content-Type` 与 `Accept` 分别描述什么？
2. 为什么删除数据不能放在 `GET` 请求中？
3. 创建资源成功时，`201` 与 `200` 的表达重点有什么不同？
4. `400` 和 `500` 的责任边界有什么差别？
5. 为什么错误响应需要稳定的 `code`，而不应只依赖 `message`？
6. REST 的无状态约束为什么不等于服务器不能使用数据库？
7. 资源与资源的 JSON 表示有什么区别？
8. `no-cache` 与 `no-store` 的含义有什么不同？
9. 条件更新中的 `If-Match` 可以避免哪类覆盖问题？

参考答案：`Content-Type` 说明发送内容的格式，`Accept` 说明期望的响应格式；`GET` 在语义上应当安全；`201` 明确表示新资源已创建；`400` 通常是请求不符合约定，`500` 是服务器未预期地失败；`message` 可能因语言或文案变化，`code` 更适合程序判断。无状态要求单次请求包含必要上下文，数据库保存的是资源与业务状态；资源是被识别的业务概念，JSON 只是它的一种表示；`no-cache` 要求复用前验证，`no-store` 禁止存储；`If-Match` 可以检测客户端基于旧版本提交的更新，避免静默覆盖较新的修改。
