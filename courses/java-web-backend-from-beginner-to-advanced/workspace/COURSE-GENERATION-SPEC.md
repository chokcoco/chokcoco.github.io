# Course Generation Spec: Java Web 后端入门到进阶

状态：第二模块深度返工 v0.5。

## Course shape

- 保持 36 章和四个模块，不设置单章字数、时长或全课总时长上限
- 章节体量由知识点本身决定，以“概念、语法、原理、用法和边界已经讲清，并有可验证练习”为停止条件
- 主线为“运行 Java → 掌握语言基础 → 使用标准库 → 理解 HTTP 与 Spring Boot → 完成数据与交付闭环 → 学习进阶机制”
- 订单、库存等业务场景只在基础概念和基本用法讲完后出现，用于迁移知识，不承担开场引入职责

## Module allocation

| 模块 | 章节 | 章节数 | 模块目标 |
| --- | ---: | ---: | --- |
| 环境介绍与准备 | 1–2 | 2 | 说明学习目标、系统地图，并建立可复现的 JDK 17、IDE、Maven 环境 |
| Java 语言及标准库基础 | 3–16 | 14 | 从程序结构和语法起步，掌握类型、对象模型、集合、异常与 Stream |
| 后端开发基础 | 17–29 | 13 | 从 HTTP 基础进入 Spring Boot、数据持久化、测试与 Docker |
| Java 进阶 | 30–36 | 7 | 在前置知识完整的基础上学习安全、缓存、消息、并发与 JVM |

## Version policy

- 所有主线代码以 JDK 17 编译，不启用 preview feature
- JDK 17 已稳定提供 record、sealed class 和 `instanceof` 模式匹配，可以进入主线，但必须在首次使用前解释语法
- switch 模式匹配、record pattern、虚拟线程等 JDK 21 稳定能力只放入版本差异或进阶旁路
- Spring Boot 主线锁定 3.x；框架默认行为或异常类型要标出实际核验的小版本

## Learning stages

1. 工具就绪（1–2）：能理解课程目标与系统地图，独立运行最小 Java 程序，核验终端、IDE 和 Maven 的 JDK
2. 语法入门（3–7）：能写变量、表达式、控制流和简单方法，理解类型、引用与相等性
3. 面向对象与标准库（8–16）：能使用类、接口、异常、泛型、集合、Stream、文件与时间 API
4. Web 入门（17–23）：先理解 HTTP，再理解 Spring Boot 请求处理、分层、校验、异常和 SQL
5. 数据与交付（24–29）：能完成数据访问、事务、测试、外部调用处理和 Docker 运行
6. 进阶机制（30–36）：在基础能力之上学习安全、缓存、消息、并发、JVM 与诊断

## Zero-basis chapter contract

每章都要先做前置知识审计。正文不得假定学习者已经掌握后续章节才会讲到的语法、API 或框架概念。

标题和内容遵守以下约束：

- 章标题、目录标题使用直叙句，不使用疑问句或悬念式标题
- 开头先说明本章学习对象及必要前置，不用未解释的业务 bug、大段代码或架构故障开场
- 语言与概念章节先解释“它是什么”，再讲语法形式、基本原理和基本用法；学习者能运行最小示例后，才进入进阶点、易错点和后端应用场景
- What / Why / How 仍是作者侧检查维度，不在学生页面机械展示
- 不把“概念 → 最小示例 → 原理 → 边界 → 练习”作为通用模板。章节结构由教学任务决定，具体选择见下一节
- 常见误区之后放自然的小结，再进入快速自测；章末保留公共参考文献
- 学生可见正文的教学二级标题统一使用中文序号，三级及更深标题使用递进编号；章末固定使用未编号的“参考文献”标题以保持生成器入口一致。环境章节仍按操作任务组织，不因此套用语言概念课结构。

## Chapter structure policy

| 教学任务类型 | 适用章节 | 建议正文组织 | 不应做的事 |
| --- | --- | --- | --- |
| 环境落地 | 1–2 | 完成条件 → 操作路径 → 每步结果 → 版本核验 → 排错 → 验收记录 | 用抽象代码示例代替安装和验证 |
| 语言与概念 | 3–16、35 | 概念 → 语法/API → 小示例 → 机制 → 边界 → 练习 | 跳过基本概念，直接给业务代码 |
| 协议、框架与架构 | 17–22、30–31 | 观察请求或输入输出 → 组件职责 → 最小闭环 → 配置 → 常见误配 → 验收 | 把注解或配置清单当作解释 |
| 数据与交付 | 23–29、32–33 | 要解决的问题 → 最小闭环 → 操作/配置 → 失败模式 → 运行或查询证据 | 只讲名词，不给可观察结果 |
| 并发与诊断 | 34、36 | 现象或症状 → 运行模型 → 复现/取证 → 判断与处理 → 复盘 | 先给参数表或结论，不解释证据 |

每章在写作前必须有一份“教学任务类型 + 建议正文组织”的计划。二级标题要服务该章的内容，不能批量复用同一组模板标题。

## Local environment and operating-system coverage

- 面向本课程的环境、IDE、构建、容器、路径和命令章节，macOS 与 Windows 都是正式支持的平台；不能将其中一个平台视为附注。
- 每项首次安装的工具必须先用日常语言解释其职责，再给出官方获取渠道、系统与 CPU 架构选择、安装步骤、版本核验、最小成功结果和常见失败的排查入口。
- macOS 与 Windows 的命令、安装包格式、环境变量界面和路径写法必须分开呈现。不能让学习者把 `/usr/...`、`.dmg`、Homebrew 命令用于 Windows，也不能让其把 `C:\...`、`.msi`、系统环境变量步骤用于 macOS。
- 课程默认推荐的 IDE 必须说明 IDE 的含义、为什么推荐、替代工具的适用边界，以及该 IDE 在两种系统上的安装和项目 JDK 选择路径。
- 只有在某个后续工具确实不支持一方平台时，才允许只提供单一系统路径；正文必须明确标注限制并给出可行替代方案。

## Action links and beginner tone

- 一般事实来源仍在章末“参考文献”汇总；但学习者必须访问才能完成当前操作的官方地址，例如软件下载页、安装说明、在线工具和控制台入口，必须直接嵌入对应操作句，不让读者回到文末查找。
- 面向零基础读者时，解释术语、目的、判断标准和必要的风险；删除作者的推进话术、对读者能力的揣测、重复的操作提醒和“不要把 A 用到 B”式内心独白。
- 只有当一个错误会真实阻碍当前操作时，才用短句说明其原因和修复动作。警告必须紧挨相关步骤，不能代替正常说明。

## Code dependency gate

- 每段代码出现前，检查其中的关键语法是否已在前文讲过
- 如果当前知识必须依赖尚未系统讲解的语法，先用一个小节补足最小前置，再展示代码；后续章节再做完整展开
- 第一段示例只验证一个概念。不要在学习者还不认识变量、方法、类或注解时，同时引入多层对象、集合链、业务规则和框架调用
- 代码块只保留理解当前概念所需的行，并给生词和关键语句加注释；完整示例放在基础讲解之后
- 示例由短到长：表达式或声明 → 可运行的小程序 → 组合示例 → 应用场景
- 不能用“读者有前端经验”作为跳过 Java 基础语法的理由

## 全课程防缩水基准与九步完整性审查

第 6 章是语言及标准库章节的解释粒度与组织方式参考，不以它的目录复制其他正文类型。除课程开篇、模块过渡与总结类正文外，各章以约 12,000 个 Unicode 有效字符作为防缩水基准：低于基准默认进入补充或说明，高于基准也不能自动判定完整。

有效字符统计用于发现异常短小章节：统计标题、解释正文、有效表格、必要代码注释、练习和答案；不把 Markdown 标记、URL 和参考文献地址当作有效内容。代码可以记录在章节总量中，但大段代码不能代替解释。知识范围确实较小的章节可以低于建议值，必须记录原因并通过其余检查。

每章按以下九步审查：

1. **体量初筛**：记录有效字符数，判断是否存在明显压缩；不以数字直接判定通过。
2. **知识范围**：核对核心对象、必要前置、语法/API、基本操作、工作机制、应用方式、边界、后续依赖和本章不展开项。
3. **零基础前置**：逐段审计术语与代码；越级内容必须补前置、简化或后移。
4. **教学层次**：语言章节检查基础概念、语法/API、基本用法、工作机制、应用组合与边界易错；协议、框架、实践与诊断章节按其正文类型检查完整链路与完成证据，不强制套用语言课六层目录。
5. **示例代码**：检查前置、单一目标、必要注释与 JDK 17 可运行性。带控制台输出的示例用相邻注释标明预期结果，多步骤代码在关键动作处解释。
6. **学习闭环**：至少用概念检查、代码阅读、代码修改、独立练习、自测或完成标准中的合适组合检验学习成果。
7. **篇幅有效性**：删除重复结论、无关场景、代码堆积、表格复述和过长总结；体量不足时补缺失知识层，不扩写空话。
8. **跨章关系**：检查前置、重复、矛盾、术语、命名、示例复用和难度曲线。
9. **状态判定**：以证据标记“完整正文”“需要补充”或“结构性返工”。只有九步整体通过，才能标记完整正文。

第二模块二级标题使用中文序号，如“## 一、……”；三级标题使用对应递进编号，如“### 1.1 ……”。编号服务长文导航，标题仍按各章知识点自然设计。

## Core route and weighting

核心章为 4、5、6、7、8、10、11、12、13、14、17、18、20、21、23、24、25、26、27、30、31、32、33、34、35、36。核心代表需要更充分的解释和练习，不代表可以压缩其他章节的基础内容。

## Backend architecture policy

- 第 17 章先解释 HTTP 请求与响应，第 18 章再进入 Spring Boot 注解和请求路径
- 主案例采用 Controller、Application/Service、Repository/Mapper 的简化分层；相关术语先定义后使用
- 参数校验先讲输入、约束、验证结果和异常，再讲 `@Valid`、嵌套校验与全局错误契约
- 数据库章节先讲表、行、键、SQL 和索引，再引入 MyBatis-Plus 与 JPA
- Security 先讲过滤器链、认证、授权和密码存储，再实现 JWT
- Redis 与 MQ 先讲各自解决的问题和基本模型，再讨论失败、一致性与可靠性

## Data, MQ and identity decisions

- MyBatis-Plus 与 JPA 各占一章，使用同一小需求对照，不在同一业务代码中混用
- MySQL 与 PostgreSQL 的共同 SQL 能力放在主线，产品差异集中说明
- Kafka 完成最小生产、消费、消费组、重复消息与失败处理演示；RabbitMQ 与 RocketMQ只做模型对照
- JWT 同时解释自签发与 OAuth2 Resource Server 路线，重点比较责任边界，不建设授权服务器

## Student-visible style

- 采用清楚的多级标题和递进编号，参考 `课程风格参考_6.md` 的信息组织方式
- 吸收 `Java 变量与数据类型.md`、`Java 反射概述.md` 中“先概念、后基本操作、再完整示例”的节奏，但不沿用其中的过时 API、不准确结论、宣传语气和大段未拆解代码
- 写给 Java 零基础学习者。术语第一次出现时先用日常语言解释，再给正式定义；首次出现使用 `[[术语]]`
- 不使用表情符号、宣传式开场、公式化金句、装饰性粗体和聊天收尾
- 类比只帮助建立第一印象，随后必须回到 Java 的真实语义和边界

## Source policy

- Java 语言与 JVM：JLS、JVMS、Java 17 API、JEP、dev.java
- Spring：课程锁定版本的 Spring Boot、Framework、Security 官方文档
- 数据、中间件与部署：数据库、MyBatis、JPA、Redis、Kafka 和 Docker 官方文档
- 用户提供的教程用于教学顺序、例子粒度和易错点线索；技术事实仍由官方资料校准

## Deliverables

- 36 章 Markdown 事实源和静态 HTML
- JDK 17 + Maven + Spring Boot 示例工程
- 语言、集合、REST、数据、安全、缓存、MQ 与 JVM 速查资料
- 章节练习、参考答案和可执行测试
- 确定性质检、语义审校和浏览器体验报告

## Quality gates

正式生成与返工均按以下顺序执行：构建与渲染 → humanizer-zh → `validate-course.mjs` → `course-rubric.md` 语义审校 → Maven 测试与服务启动 → 浏览器验证。任何正文修改都会使旧报告失效。

第 3 章起默认以连续 3–4 章为一个生成与验收批次。每批同时完成 Markdown、HTML、Humanizer、确定性检查、批次评价量规和浏览器体验审校。批次审校必须读取前一章正文以及后一章蓝图或草稿，检查批次边界的前置、术语、示例和导航衔接。本轮批次为第 3–6 章，scope 使用 `03-06`。

语义审校增加三项课程专用检查：

1. 首段是否出现未解释的代码、业务场景或术语。
2. 代码是否依赖后续章节才会讲到的语法；若依赖，本章是否已补足最小前置。
3. 是否先讲基础概念和基本用法，再进入进阶机制与应用场景。
4. 除课程开篇、模块过渡与总结类正文外，章节是否达到约 12,000 有效字符的防缩水建议值；低于建议值是否有合理说明，所有章节是否通过九步完整性审查且没有填充式扩写。
5. 长章是否使用中文序号二级标题与递进编号三级标题；代码的关键步骤和可观察输出是否有就近注释。

## Publishing

默认人工审核，不自动发布。正文生成后仍不注册到 `courses/catalog.json`，由用户决定是否公开到平台目录。
