为什么需要 RAG：从闭卷回答到有据可查

如果只用一句话说清这门课，我会这样说：我们要做一个小型企业知识库，让模型回答客户问题前先查资料，并且把证据交出来。第一章先不碰复杂框架。先把问题想明白：为什么普通聊天不够，为什么搜索也不够，RAG 到底补上了哪一块。

先把这个麻烦看完整

假设一家科技公司有三类资料：产品 FAQ、退款制度、历史客诉工单。客服问模型：“用户买了年度会员，使用 3 天后申请退款，应该怎么处理？”

如果只是普通聊天，模型大概率会给一个听上去挺像样的答案。它可能会说“可以退款”“请联系客服”“一般 3 到 7 个工作日到账”。这些话不一定错，但问题在于：公司制度里有没有 3 天这个条件？优惠券要不要扣减？客服能不能承诺到账时间？如果用户已经用了会员权益，是否要转二线复核？

RAG 要解决的不是“让模型说得更漂亮”。它要解决的是：回答之前，系统先把相关资料找出来，回答之后，还能让人看到它依据了哪几段资料。你可以把它理解成把模型从闭卷考试拉回开卷考试，而且试卷旁边要写出处。

RAG 的核心是证据链，不是向量数据库

[[RAG]] 的完整名字是 Retrieval-Augmented Generation。Retrieval 是检索，Generation 是生成。先找资料，再回答。这句话很短，但里面有一个容易被忽略的前提：资料必须能被系统找到，也必须能被人核对。

很多教程一上来就讲向量数据库。向量库当然重要，后面我们也会认真讲。但企业里更早遇到的问题往往不是“向量库选哪个”，而是资料有没有来源、版本、权限、负责人。没有这些东西，系统即使找到了片段，也很难判断它能不能用。

所以我们从第一章就定一个数据形状。它还不是最终实现，只是把“证据链”这件事落在字段上：每个片段至少要知道自己来自哪里、属于哪一节、什么版本、谁能看。

Prompt、搜索、RAG、微调，其实在管四件不同的事

初学者很容易把它们排成一条升级路线：Prompt 不行就 RAG，RAG 不行就微调。这样想省事，但会把项目带偏。

Prompt 更像答题规范，解决“怎么答”的问题。搜索解决“在哪里找”的问题，但它不会自动替你组织成带引用的客服回复。RAG 把找资料、组上下文、生成答案连成一条链。微调更重，它适合长期改变模型的行为习惯，比如固定标注风格、固定分类口径，而不是追每周更新的退款制度。

场景优先方案原因
只是想让回答格式稳定Prompt事实已经在输入里，先约束表达
想查公开网页资料搜索人工确认或简单摘要可能够用
要回答企业制度、FAQ、工单RAG资料会更新，还需要引用和权限
要模型长期学会固定标注风格微调目标是改变模型行为，不是补资料

这张表的用法很简单：先问事实从哪里来，再问资料会不会变，最后问答案是否需要引用。如果答案依赖企业内部资料，而且资料会更新，RAG 通常就是更自然的起点。

课程项目先搭出轮廓，不急着堆工具

这门课后面只围绕一个项目：科技公司内部知识库 / 客诉答疑 RAG。我们会先用虚构资料，但资料结构按真实企业来：产品说明、退款制度、历史工单、测试用例模板、权限字段、版本号。

第一阶段不接真实模型，也不接向量数据库。听起来像退一步，其实是在保护学习节奏。先把目录、输入、输出和验收方式站住，再替换更复杂的部件。否则一出错，你会分不清是 API、检索、分块、Prompt 还是数据结构的问题。

让 AI 生码可以，但要给它边界

后面的实践会借助 Codex 或 Claude Code。这里有个小经验：不要让 AI “帮我写一个 RAG 系统”。这个指令太大，它会把框架、依赖、文件结构和业务规则混在一起，生成一堆看似完整但很难验收的代码。

更好的做法是把目标、阶段、数据字段、拒答规则和验收命令写清楚。你让 AI 写代码，但项目的边界由你来定。

先用 mock 模型跑通链路

第一章的代码故意“不智能”。我们先写一个 mock LLM：有资料就基于第一段资料回答，没资料就拒答。它的价值不是效果，而是逼我们把返回结构想清楚。

注意返回里的 citations、missing_info 和 confidence。这些字段后面会变得很重要。企业知识库不是只返回一句话，它还要告诉你这句话靠不靠谱，靠的是哪份资料。

快速自测

RAG 最先解决的是哪类问题？

RAG 的核心是把外部资料接入回答流程，尤其适合企业私有资料和频繁更新的知识。

只想让模型按 JSON 格式回答，优先改什么？

格式约束通常先从 Prompt 做起，不要一上来就搭 RAG 或微调。

企业 RAG 为什么要保存引用？

引用让答案可追溯。答错时也能追查是资料、检索还是生成出了问题。

收个口

这一章先把地基打好：RAG 不是一个更会聊天的按钮，而是一条证据链。它让模型先看资料，再组织答案。下一章开始，我们会从空目录把项目搭出来，并把“资料应该长什么样”这件事定下来。
