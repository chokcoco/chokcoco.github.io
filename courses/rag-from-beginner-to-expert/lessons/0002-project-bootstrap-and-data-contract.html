从空目录开始：项目骨架、资料格式和 AI 生码边界

第一章我们把 RAG 的位置说清楚了：它不是聊天包装，而是一条证据链。第二章开始动手。别急着上向量库，也别急着让 AI 一次写完系统。先把项目骨架和数据契约定住。契约定得清楚，后面让 Codex 或 Claude Code 写代码时，才不会每一章都改一遍数据结构。

建立项目目录准备 mock 模型适配器设计 documents.jsonl写可复用的生码提示词

这一章要交付什么

  • 建立项目目录
  • 准备 mock 模型适配器
  • 设计 documents.jsonl
  • 写可复用的生码提示词

先别急着“智能”，先让资料有规矩

很多 RAG 项目卡住，不是因为模型不够强，而是因为资料进入系统时太随意。今天一个 PDF，明天一个飞书页面，后天又来一份客服 Excel。只要它们进系统后都变成一团纯文本，后面就很难做权限、引用和评测。

第二章先定一个朴素但够用的数据契约：原始资料放在 data/raw，处理结果写到 data/processed/documents.jsonl。每条文档必须带来源、标题、章节、版本、权限和更新时间。你可以把这些字段理解成资料的身份证。没有身份证，系统就算把它找回来了，也不知道它是谁。

README 不是摆设，它是项目的第一份契约

对初学者来说，README 最容易被忽略。可是在 AI 生码项目里，它反而很重要。你每次让 Codex 或 Claude Code 改代码，它都会根据当前文件理解项目边界。README 写得清楚，AI 就少一点自由发挥。

这里的 README 不写宏大愿景，只写当前边界、目录约定和命令。等后面接真实模型、向量检索、前端页面时，再逐步加。

原始资料也要稍微整理一下

RAG 不喜欢来路不明的资料。我们先用 Markdown 做原始资料，因为它对人友好，也方便后面按标题和段落切分。文件开头的 frontmatter 保存元数据，正文保存制度内容。

注意这里没有写成客服话术，而是写成制度事实。话术可以后面在生成阶段处理；知识库里最好先保存相对稳定、可引用的事实。

解析脚本要“挑剔”一点

如果资料缺字段，脚本应该直接报错，而不是默默吞掉。课程前期你可能会觉得报错烦，但企业知识库最怕静默失败。静默失败的意思是：系统表面跑通了，实际上资料没入库、字段错位、权限丢失，最后模型还给了一个很自信的答案。

下面这个解析器很短，只做一件事：把 Markdown 的 frontmatter 和正文拆开。它不追求兼容所有格式，先服务我们的学习项目。

越早校验，越少返工

数据契约不能只写在文档里，还要写进代码。assertDocumentShape 会检查每条文档是否带齐字段。后面做 chunk、检索、引用时，它们都默认这些字段存在。

这也是给 AI 生码工具加护栏。AI 可以继续写 ingest、chunk、retrieve，但只要输出不符合 schema，脚本就会拦下来。

把 raw 资料变成 documents.jsonl

现在可以写第一条真正的数据流水线了。它读取 data/raw 下的 Markdown，解析元数据，校验字段，然后写出 JSONL。JSONL 的好处是简单：一行一条记录，用编辑器也能看，用脚本也好处理。

这一章只做到 document 级别，不做分块。分块会在第 4 章展开。先把资料从“文件”变成“可检查的记录”，这是更稳的顺序。

练一下

新增一份 `product-faq.md`，照着 `refund-policy.md` 写 frontmatter 和正文。再运行 `npm run ingest`。如果脚本报缺字段，不要绕过校验，先把资料补完整。

快速自测

为什么第二章先定数据契约？

RAG 后续每个环节都会消费这些字段。契约不稳，检索、引用、权限和评测都会跟着返工。

permission 字段主要服务什么？

企业知识库不能只看相关性，还要看用户有没有资格看到这段资料。

documents.jsonl 为什么适合做中间产物？

JSONL 一行一条记录，适合调试、追加和被后续脚本继续处理。

这一章收在这里

第二章的重点不是写很多代码，而是把“资料进入系统时必须长什么样”定下来。RAG 后面的检索、引用、权限、评测都吃这份结构。下一章会继续往前走：真实企业资料通常不干净，我们要开始处理噪声、重复、版本和权限。
