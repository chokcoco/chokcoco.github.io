资料进入知识库：加载、清洗、元数据和版本

第二章我们把文档契约定住了。现在要面对更真实的一面：企业资料很少乖乖躺在一个干净文件夹里。FAQ 有重复答案，客诉工单带口语和情绪，制度文档有版本差异，某些内容还带权限。第 3 章不急着检索，先把资料治理到“可以放心入库”的程度。

真实资料进来时，通常没那么体面

如果只看教程里的 demo，资料总是干净的：一段制度、一条 FAQ、一个标准答案。可真到公司里，资料常常长得很别扭。同一条退款规则在 FAQ 里写了一遍，在客服手册里又写了一遍；历史工单里有有用结论，也夹着用户情绪、客服口头承诺和一些已经过期的处理办法。

这一步如果偷懒，后面检索会很难看。模型可能找到过期规则，也可能把只允许主管看的客诉处理记录拿去回答一线客服。第 3 章做的事情不花哨：把原始资料整理成可靠的入库物料。它更像厨房备菜，刀工不好，后面再会炒也救不回来。

先补两份原始资料。一个是 FAQ，一个是历史工单摘要。你会看到它们的权限不同：FAQ 给客服可见，客诉工单摘要只给主管角色可见。这不是形式主义，后面的权限过滤会直接吃这个字段。

清洗不是越狠越好

初学者容易把“数据清洗”理解成删得越干净越好。这个想法有点危险。RAG 里的资料不是训练集，很多结构本身就有意义：Markdown 标题告诉你章节，列表告诉你步骤，空行告诉你段落边界。清洗时把这些都抹平，后面分块就只能靠字数硬切。

所以这里用一版很克制的清洗：统一换行，去掉行尾空格，压缩过多空行，删除明确标注为 private 的内部批注。其他东西先保留。遇到不确定的内容，不要让脚本自作聪明。

元数据是检索时的方向盘

很多人第一次做 RAG，只保存 text。一开始确实能跑，问一句退费政策，系统能找回几段文字。可只要项目稍微真实一点，问题就来了：用户是什么角色？这段资料是不是最新版？答案引用的是 FAQ 还是历史工单？出了错找谁维护？

这些问题都不是模型能凭空解决的。元数据要在入库前就补齐，而且缺字段要直接报错。这里我们先把权限范围收窄到几个明确值，避免后面出现 Support、support_user、客服 这类混乱写法。

重复资料会让模型变得“很有信心地摇摆”

重复内容不只是浪费空间。更麻烦的是，两个重复片段如果版本不同，检索时可能一起被找出来。模型看到两个相似但不完全一致的规则，就会开始混合答案。用户看起来得到了一句完整回复，实际上是两份资料拼出来的。

第 3 章先做一个简单指纹去重。它不追求识别所有近似重复，只处理“内容几乎一样”的情况。更复杂的近似去重可以放到后面迭代，不要一开始就把项目做成数据治理平台。

把导入过程写成能复盘的流水线

现在把清洗、元数据校验和去重串起来。这里有个小习惯值得保留：导入脚本不要只输出 documents.jsonl，还要输出 import-report.json。以后系统答错了，你至少知道当时入库了几份资料、跳过了哪些重复项。

这份报告不面向用户，面向开发者和课程练习。它让“资料有没有正确进入系统”这件事可以被检查，而不是靠感觉。

练一下

故意复制一份 `refund-policy.md`，改个文件名但不改正文。运行 `npm run ingest`，看看 `import-report.json` 是否能识别重复。再把其中一份的 `permission` 改成不存在的值，确认脚本会报错。

快速自测

为什么不把历史工单原文直接塞进知识库？

历史工单常有情绪、口语和个人信息，适合整理成摘要后再入库。

permission 字段在后面最直接服务什么？

检索时不仅要看相关性，还要按用户角色过滤可见资料。

import-report.json 的价值是什么？

导入报告能告诉你输入多少、保留多少、跳过哪些重复资料。
