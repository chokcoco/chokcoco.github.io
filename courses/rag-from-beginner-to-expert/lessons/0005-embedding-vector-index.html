Embedding 和向量索引：把文字放进可搜索的语义空间

上一章，我们已经把文档切成了 chunk。现在问题来了：用户不会总按资料原文提问。资料里写的是“优惠券抵扣部分需要扣减”，用户问的可能是“用了券还能退全款吗”。只靠关键词，系统有时能碰上，有时就错过。第 5 章要做的事，就是把 chunk 和问题都变成向量，再用相似度把它们找回来。

解释向量相似度理解同一模型空间搭建本地向量索引设计可替换接口

这一章要把“能切”推进到“能搜”

  • 解释向量相似度
  • 理解同一模型空间
  • 搭建本地向量索引
  • 设计可替换接口

先看关键词检索的尴尬时刻

假设 chunk 里写着：“已经使用优惠券的订单，需要按优惠券规则扣减。”用户问：“用了券还能退全款吗？”这两个句子明明在问同一件事，但关键词重合并不多。你要是只搜“全款”，可能什么也找不到；只搜“券”，又可能把发票、活动、优惠说明都拉回来。

这就是 [[embedding]] 出场的地方。它不是把文字翻译成另一种语言，而是把文字变成一串数字。数字本身人类读不懂，但可以计算距离。两个句子意思越接近，向量方向通常越接近。先别急着觉得玄，我们用最小代码把这个过程拆开。

理解向量，有一个小技巧

可以把每个 chunk 想成地图上的一个点。问“优惠券退款”，它会落在退款制度附近；问“发票抬头填错”，它会落在发票 FAQ 附近。向量检索做的事，就是把用户问题也放到这张地图上，然后找离它最近的几个点。

这张地图有两个规矩要记住。第一，chunk 和 query 必须用同一个 embedding 模型生成，地图才是同一张。第二，向量库里不能只存 embedding，还要带回原文和元数据。否则你只知道“这个点很近”，却不知道它来自哪里、谁能看、能不能引用。

技巧一：先做教学版 embedding，别一上来接 API

这一章我们先写一个很朴素的 mock embedding。它会把文本拆成相邻字符窗口，再把这些窗口塞进固定长度的数组。这个算法不聪明，甚至有点粗糙。但它有一个好处：不需要 API key，也不会因为网络或额度问题打断学习。

这里要说清楚：mock embedding 不是拿来证明检索效果的，它是拿来证明链路形状的。只要链路形状对了，后面把模型换成真实 embedding，项目结构不用推翻。

技巧二：相似度不是“包含关系”，而是方向关系

关键词搜索像是在问：“你有没有这个词？”向量搜索更像是在问：“你们的方向像不像？”这也是为什么 RAG 里常见余弦相似度。它不太关心向量有多长，更关心两个向量夹角接近不接近。

下面的函数很短，但很值得手敲一遍。写完它，你会发现“语义检索”至少在工程上并不神秘：生成向量，算分，排序。真正难的是数据、模型和业务边界，不是这几行数学。

技巧三：向量索引不是只存一串数字

现在把上一章的 chunks.jsonl 变成 vectors.jsonl。请注意，每条记录除了 embedding，还保留了 title、source、section_path、permission、version 和 text。很多初学者会把向量索引理解成“向量数据库里只要有向量就行”，这会把后面的引用和权限过滤全弄丢。

这里仍然用 JSONL 做本地索引。它慢，但是透明。等第 6 章再谈真正的检索策略时，你已经知道每条向量记录到底应该长什么样。

技巧四：先全量扫描，确认排序真的发生了

如果只有几十个 chunk，没必要一开始就搬出向量数据库。全量扫描慢归慢，但它能把逻辑摊开：问题向量生成、权限过滤、相似度计算、按分数排序、取前几个结果。等这些都对了，再换成 HNSW、IVF 或托管向量库，心里会踏实得多。

这段代码里还有一个小细节：权限过滤放在排序前。因为用户没有权限看的片段，就算再相似，也不应该进入候选列表。

真实 embedding 怎么接？只替换 adapter

等你要接真实模型时，不要把整个索引脚本重写一遍。保留 build-vector-index.js 的主体，把 embedTextMock 换成真实 adapter。这个边界很重要。RAG 项目里，模型供应商、向量库、重排模型都可能替换，主链路不能跟着乱。

练一下

运行 `npm run check:index`。然后分别搜索 `年度会员退款`、`用了券还能退全款吗`、`发票抬头错了怎么办`。观察分数和命中片段：mock embedding 能让链路跑通，但它并不理解真正语义，这个差距要记住。

快速自测

为什么查询和 chunk 要用同一个 embedding 模型？

不同模型生成的向量不在同一个语义空间里，拿来直接比较没有意义。

mock embedding 在本课程里主要用来做什么？

mock 版本不追求真实效果，它帮助我们先把索引、搜索、过滤和验收跑通。

向量索引里为什么还要保留 text 和 metadata？

搜索命中的是向量记录，但回答和权限判断仍然要依赖原文、来源和权限字段。

这一章最后再捋一下

第 5 章把 chunk 推进到了可搜索的语义空间。我们没有一上来接真实向量库，而是先用 mock embedding 看清楚四件事：同一模型空间、向量归一化、相似度排序、metadata 不能丢。下一章会继续往前走，把关键词检索、语义检索和权限过滤合在一起，做出第一版真正能用的 retrieve。
