# Course Generation Spec

## Scope

生成“RAG 从入门到精通”课程。当前阶段交付完整 12 章框架和第 1-7 章样张。

## Style

- 用中文写给 0 基础或轻编程经验学习者。
- 不使用固定的 What / Why / How 小标题模板。可以在脑内按这个逻辑组织，但正文要像人写的技术文章。
- 风格参考 assets/课程风格_wenjie.md 和 assets/课程风格_wenjie_2.md：不要急着下定义，先把一个真实麻烦讲完整，再解释为什么会失败，最后给代码和验收办法。
- 第 5 章起允许按章节主题切换参考风格。Embedding 章节参考 assets/课程风格_coco.md、assets/课程风格_coco_2.md、assets/课程风格_coco_3.md：更强调“这个属性/函数到底干什么、有哪些技巧、完整代码怎么跑”。
- 从第 6 章开始重点参考 assets/课程风格参考_6.md：先建立概念，再讲技术发展和原理，最后讲应用、痛点和总结。
- 第 6 章开始大幅减少代码量，不再围绕一个贯穿项目展开。可以使用流程图、对比表、短伪代码、手动练习替代项目代码。
- 不把每个重点都写成“一个标题 + 一段正文”。章节内部要根据内容变化排版：连续叙事、反例、表格、流程图、代码、练习可以穿插使用。
- 每个主题要讲完整，避免正文戛然而止。至少讲清：它是什么、为什么需要它、在本项目里怎么落地、容易误解在哪里。
- 每章需要有本章参考资料。
- 每章至少有一个知识点讲深，配图、代码、例子或反例。
- 第 1-5 章保留已有代码型样张；第 6 章以后代码块只在确有必要时出现，优先使用图、表、概念拆解和手动练习。
- 如果出现代码块，必须单列显示，并解释它服务的概念，不把章节写成项目开发教程。
- 代码块必须有本地高亮效果，不依赖外部 CDN 才能成立。
- 代码需要必要注释，解释业务边界、数据契约、错误处理和验收意图。

## Final Practice Spine

最后 1-2 章再集中进入“科技公司内部知识库 / 客诉答疑 RAG”实践。前面的概念章节只用轻量案例解释技术，不持续推进项目代码。最终实践包含：

- data/raw 企业资料
- ingest / chunk / index / retrieve / answer / citations / eval
- mock adapter 和 OpenAI-compatible adapter
- 本地 API
- 最小前端页面
- 评测集和项目报告

## Source Policy

社区内容要大量参考，尤其是 RESOURCES.md 的 Wisdom 部分。使用前要用论文、官方文档或其他可信资料校准关键事实。
