# Mission: RAG 从入门到精通

课程主要内容包含：

- RAG技术基础：深入浅出地介绍RAG的核心概念、技术原理和应用场景
- 数据处理全流程：从数据加载、清洗到文本分块的完整数据准备流程
- 索引构建与优化：向量嵌入、多模态嵌入、向量数据库构建及索引优化技术
- 检索技术进阶：混合检索、查询构建、Text2SQL等高级检索技术的简单介绍和了解
- 生成集成与评估：格式化生成、系统评估与优化方法的简单介绍和了解
- 项目实战：从基础到进阶的完整RAG应用开发实践


## Why

这门课要帮助 0 基础或有简单编程经验的人员理解为什么 AI 工程里会出现 RAG，并能把 RAG 从一个抽象名词转化为可理解、可操作、可评估的企业知识库方案。学习者完成课程后，应能看懂 RAG 系统的核心流程，知道每个组件解决什么问题，并能通过教学文档，一步一步基于一个已有的 AI 模型（如 Codex 或者 Claude Code），基于已有的一些文档，实现一个简单的 RAG 知识库，同时提供企业级的整体解决方案，为后续转向 AI Agent 工程师、全栈开发工程师或独立 AI 开发者打基础。

## Success looks like
- 学习者能用生活化语言解释 RAG 解决的问题：让 AI 在回答前先查可信资料。
- 学习者能了解 RAG 的基本流程：资料整理、切分、索引、检索、生成、引用、评估。
- 学习者能区分 prompt、embedding、向量库、检索、reranking、上下文窗口、幻觉、引用这些概念。
- 学习者能判断一个业务场景是否适合用 RAG，而不是盲目微调模型或只写 prompt。
- 学习者能描述企业级知识库需要考虑的权限、数据更新、评测、成本、监控和安全风险。
- 学习者能完成一个无代码或低代码练习：上传企业资料，配置知识库，提出问题，检查引用和答案质量。
- 学习者能完成一个 1-2 天实践项目：科技公司客诉答疑、业务测试用例生成或代码库助手三选一。

## Constraints
- 面向 0 基础或轻编程经验学习者，过程中逐渐提供代码或者伪代码（语言提供 JAVASCRIPT 和 JAVA 两个版本）
- 课程语言使用中文，少用工程黑话；必要术语要先定义，再使用。
- 每节 lesson 包含 2-4 个紧凑目标，并包含即时反馈练习，每个知识点除了整体的总结，还需要有详细的描述，每一章节学习耗时控制在 1 个小时左右，整体 12 个章节；
- 每一章节至少有一个知识点需要深入到细节，完全展开讲解，包括但不限于深入讲模型训练、微调、Transformer 数学细节，处理清理细节、检索细节等等；
- 课程最终形态应兼容当前 teach workspace：`lessons/*.html`、`reference/*.html`、`RESOURCES.md`、`GLOSSARY.md`。
- 课程可以讲工具和平台，但不绑定单一厂商；涉及具体工具时要说明可替换性。
- 理论学习控制在 12 个小时，实践项目控制在 1-2 天。
- 实践路线低代码优先，保留 LangChain / LlamaIndex 等可选代码扩展。

## Out of scope
- 不要求学习者在主线课程中掌握 Python、JavaScript、数据库或云部署。
- 不把课程做成某一个向量数据库或框架的产品教程。
- 不在第一版课程里实现完整 SaaS 平台、支付、用户系统或 LMS 集成。

## 课程输出原则

### 以读者为中心（Know Your Audience）

写文章前，先问自己：这篇文章是写给谁看的？

需要注意的是，本文面向初学者，难度定位为【高/中/低】类别的低难度： 因此需要通俗的比喻、详细的背景搭建，避免直接抛出高深术语。

### 严谨与准确（Accuracy is King）

技术文章最忌讳误导读者。

- 所有的代码/伪代码片段必须在干净的环境中验证一遍，或者确保来源的准确性；

- 引用官方文档或标准，避免使用“大概”、“可能”这类模糊词汇。

### 结构清晰，层层递进（Logical Flow）

遵循“问题 -> 原因 -> 解决方案 -> 总结”的经典叙事结构。

- What（是什么）： 清楚界定你要讨论的技术或问题。

- Why（为什么）： 为什么要用它？它解决了什么痛点？

- How（怎么做）： 具体实现的步骤、代码或架构设计。

## 课程实用技巧

### 让人眼前一亮的排版（排版即门面）

没有人喜欢看密密麻麻的文字堆砌。

- 巧用标题（Markdown ##, ###）： 让读者通过目录就能抓取文章核心。

- 利用加粗和列表： 提炼核心观点，突出关键配置或命令。

- 代码块规范： 务必指定语言类型（如 ````python``），并加上必要的注释。

### 善用图表与对比（一图胜千言）

- 架构图/流程图： 使用 Mermaid、Excalidraw、UML 语法 或 Visio 画出数据流向或系统架构，并且能够在 HTML\MD 等页面中进行有效渲染；

- 善用对比表： 在做技术选型（如 Redis vs. Memcached）时，用表格从性能、高可用、数据结构等维度对比，一目了然。

### 提供“可复制”的最小可行民主（MVP）

- 如果可以，给出完整的、可直接运行的代码示例（或附上 GitHub 仓库链接）。

- 不要丢三落四： 很多技术文章让人抓狂的原因是漏掉了某个关键的依赖项或配置。

### 摒弃行业黑话，多用类比

把复杂的概念生活化。例如：

把“负载均衡（Load Balancing）”比作餐厅的迎宾员，负责把客人均匀地分配到不同的桌子上，防止某个服务员累趴。

### 写好开篇与结尾

- 开篇： 2-3句话直接戳中痛点（例如：“你是否也遇到过线上内存溢出的情况？本文将带你…”），让读者产生共鸣。

- 结尾： 简单总结核心要点（Takeaways），或者提供扩展阅读的链接。
